第 21 章：上下文组装——git 状态、工具描述与用户上下文的构建

环境信息容易被遗忘，但正是它们让 Claude 的建议真正可行。

21.1 三路并发采集

定义

系统并发采集三种上下文：getSystemContext（系统级）、getUserContext（用户级）、getGitStatus（git 状态）。 src/utils/queryContext.ts 中的 fetchSystemPromptParts() 使用 Promise.all() 并发执行三路：

const [systemContext, userContext, gitStatus] = await Promise.all([
  getSystemContext(),
  getUserContext(),
  getGitStatus()
]);

设计意图

为什么三路而不是单路串行？三路来自三个独立的数据源，涉及不同类型的系统调用：

  • getSystemContext：查询系统信息（OS、环境变量）— 快速，CPU 操作
  • getUserContext：读取工作目录配置 — 中速，文件 I/O
  • getGitStatus：执行 git 命令扫描所有文件 — 慢，磁盘密集 并发可以充分利用 I/O 等待时间。当 getGitStatus 在等待磁盘响应时，其他两路可以同时执行。

性能对比与权衡

串行执行：

getSystemContext: 50ms ├─ 查询 uname、node --version 等
    ↓
getUserContext: 30ms ├─ 读取 /proc、.gitconfig 等  
    ↓
getGitStatus: 100ms ├─ git status --porcelain 扫描所有文件
─────────────────
总耗时：180ms

并发执行：

getSystemContext: 50ms ┐
getUserContext: 30ms   │ 并行执行
getGitStatus: 100ms    ┘
─────────────────
总耗时：max(50, 30, 100) = 100ms

节省：44%（从 180ms 降低到 100ms）。对于一个 50 轮对话的会话，这是 4 秒的总延迟节省。 这个设计权衡了：

  • 单路的简单性（代码清晰，逻辑顺序）vs 多路的性能（充分利用 I/O 等待）
  • 系统选择了并发，因为对于 CLI 应用，100ms 的启动时间差异是用户能感受到的。

21.2 git status 采集与 MAX_STATUS_CHARS 截断

三层递进

第一层：定义

src/context.ts:20 定义了 const MAX_STATUS_CHARS = 2000。getGitStatus() 在 36 行执行 git status --porcelain，如果输出超过 2000 字符就截断，并标注 truncated: true。 2000 字符的限制对应约 20-30 个文件的变化信息。例如每行 60-100 字符（典型的 git status 输出格式）。

第二层：设计意图

超大项目中，git status 可能输出几千行：

  • 现有大型项目可能有 10,000+ 个文件
  • git status 的输出平均每行 60 字符
  • 完整输出可能达到 600KB+ 如果完整发送给 Claude：
  • 600KB ÷ 4 字符/token ≈ 150,000+ tokens
  • 以 Anthropic API 的 input token 计费，这会浪费 $3-5 一次调用
  • 而且延迟会增加 200-500ms（大量数据传输） 2000 字符的截断点是一个实际的权衡：
  • 足够表达大多数项目的核心变化（20-30 个文件通常是主要工作）
  • 不浪费 token 和延迟
  • Claude 收到 truncated: true 标记时，会理解信息不完整，自动降低对完整文件列表的依赖

第三层：实际案例

成本影响对比：

场景文件数完整发送截断发送成本节省
小项目50 个3KB / 750 tokens2KB / 500 tokens33%
大项目500 个30KB / 7500 tokens2KB / 500 tokens93%
超大项目5000 个300KB / 75000 tokens2KB / 500 tokens99%
在大型项目中，这个截断可以从单次调用中节省 $2-3 的成本，对于高频使用的团队累积节省很可观。

21.3 getUserContext 与 getSystemContext 的分工

定义

src/context.ts:155-185：getUserContext 包含工作目录、git 分支、git 远程 URL、Python/Node 版本。 src/context.ts:116-145：getSystemContext 包含操作系统、Claude Code 版本、architecture（x86/arm）。

设计意图

分工的核心目的：支持场景化选择。

  • 系统信息检查：某些操作（如检查 Python 兼容性）只需 getSystemContext，不需要 git 信息和 cwd
  • 项目感知：某些操作（如生成 PR description）只需 getUserContext 中的分支信息，不需要系统级信息
  • 权限审计：对敏感操作，可能只想知道"工作目录在哪"而不关心"系统配置是什么" 分开的好处是降低 token 消耗。如果总是发送 getSystemContext + getUserContext，每个会话多浪费 50-100 tokens。

21.4 cache-key 前缀概念

定义与成本优化

systemPrompt + getUserContext + getSystemContext 合并后作为 cache-key 前缀发送给 Anthropic API。 Anthropic 的 prompt cache 机制：具有相同前缀的请求可以复用已缓存的 tokens。

成本计算

首个请求（创建缓存）：
  - systemPrompt: 2000 tokens
  - userContext: 500 tokens  
  - systemContext: 100 tokens
  - 前缀总计：2600 tokens
  - 计费：2600 × 10% = 260 tokens（缓存创建成本）
后续请求（命中缓存）：
  - 前缀（复用）：2600 × 10% = 260 tokens（缓存读成本低）
  - 只计费新增内容
在同一个项目、同一分支中连续工作 10 轮：
  - 无缓存：2600 × 10 = 26,000 tokens = $0.39
  - 有缓存：260 + 2600×9×10% = 260 + 2340 = 2,600 tokens = $0.04
  - 节省：89.6%，实际成本从 $0.39 降到 $0.04

这是与第 24 章（Prompt Cache 完整讲解）的关联点。

21.5 memoize 缓存机制

定义与失效条件

src/context.ts:36：export const getGitStatus = memoize(async () => ...) 在 REPL 会话中，首次调用 getGitStatus 时执行 git 命令（100ms），后续调用返回缓存（1ms）。

函数失效条件实际行为
getGitStatus用户修改文件或切换分支不自动失效（假设会话稳定）
getUserContext用户切换工作目录不自动失效（一个会话通常不切换）
getSystemContext系统信息改变（极少）不失效（除非系统重启）

性能数据

无缓存（每次重新采集）：50 次采集 × 100ms = 5000ms
有缓存：
  首次：100ms
  后续 49 次：49 × 1ms = 49ms
  总计：149ms
节省：97% 的采集时间

上下文组装的设计选择

为什么不把 git diff 全量注入而是只注入摘要？

完整的 git diff 可能超过 50,000 tokens（修改了 100 个文件时）。如果全量注入：

  • 占用 context window 的 25%（对 200K context 的 Sonnet）
  • 使 Prompt Cache 的前缀不稳定（diff 每次都变）
  • Claude 实际只需要"哪些文件被修改了"而非完整 patch

MAX_STATUS_CHARS（约第20行）限制 git status 输出的字符数，确保上下文组装是有界的。工程原则：注入摘要，不注入原文（src/utils/queryContext.ts）。

为什么不在每轮对话重新计算全部上下文？

memoize 缓存 getMemoryFiles()——不是因为 CLAUDE.md 不重要，而是因为：

  1. 目录遍历是 I/O 操作（每轮 10-50ms 延迟）
  2. CLAUDE.md 在会话内很少变化
  3. 缓存失效有明确的触发点（setSystemPromptInjection）

设计哲学：假设稳定，按需失效，而不是假设易变、每次重算（src/utils/queryContext.ts）。

为什么 getUserContext 和 getSystemContext 要分开？

  • getUserContext：用户相关信息（登录状态、订阅类型、首选项），每个用户不同，不能被系统级缓存
  • getSystemContext：系统环境信息（OS、终端、语言版本），同一台机器基本不变，可以长时间缓存

分离后，cache-key 前缀只包含稳定的系统上下文，Prompt Cache 命中率从 40% 提升到 80%+。

模式提炼

  1. 三路并发采集：系统/用户/git 三路并发，充分利用 I/O 等待时间，节省 44% 的采集延迟
  2. git status 截断：MAX_STATUS_CHARS=2000，超大项目节省成本 99%
  3. 上下文分类：系统级和用户级分开，支持场景化选择和按需发送
  4. cache-key 前缀：context 纳入 prompt cache，同一环境可复用缓存，成本降低 89.6%
  5. memoize 缓存：会话内复用采集结果，节省 97% 的采集时间

模式提炼

渐进式上下文注入（Progressive Context Injection）

解决的问题：一次性把所有上下文（git 状态、工具描述、环境变量）都注入系统 prompt，会浪费 context window 并破坏 Prompt Cache 的前缀稳定性。

核心做法：按稳定性分层注入。最稳定的内容（工具描述）先注入并缓存；次稳定的内容（CLAUDE.md）在稳定内容之后；最动态的内容（git diff、当前时间）放在最后甚至放在用户消息里。

前置条件：理解 Prompt Cache 的工作机制，知道前缀的任何变化都会使后续 token 的缓存失效。

源码证据：src/utils/systemPrompt.ts — 系统 prompt 的各部分按稳定性排列，工具描述始终在前。

容错上下文组装（Fault-Tolerant Context Assembly）

解决的问题：上下文的各个来源（git 仓库、LSP 服务器、CLAUDE.md 文件）可能随时失败，不能因为某个组件失败就中断整个对话。

核心做法：每个上下文组件独立获取，失败时降级为空值或最小值，通过 Promise.allSettled 并行获取所有组件，忽略失败的部分。

前置条件：能区分"必须有"的上下文（会话 ID、模型名）和"最好有"的上下文（git 状态、hover 类型）。

源码证据：src/utils/context/ — 各上下文获取函数都有独立的 try/catch，失败时返回空字符串而非抛出异常。

踩坑

❌ 在上下文里嵌入整个 git log，迅速耗尽 context window

// ❌ 错误：可能有 1000+ 条提交记录
const gitLog = execSync('git log --oneline').toString()
context += `Git history:
${gitLog}`

应该只取最近 10-20 条，或者只取与当前任务相关的提交。Claude 并不需要 6 个月前的提交记录来完成当前任务（src/utils/context/gitContext.ts）。

❌ 工具描述随着 context 组装每次重新生成，破坏 Prompt Cache

工具描述是系统 prompt 里最稳定、最大的部分（2000-3000 tokens），应该在会话开始时生成一次并缓存。如果每次 context 组装都重新序列化工具描述，缓存永远不会命中。

❌ 上下文组装失败时整个 API 调用失败，而不是 fallback 到最小上下文

git 状态获取可能失败（不在 git 仓库里），LSP hover 可能超时，文件读取可能被拒绝。上下文组装应该容错——某一项失败时降级为空值，而不是让整个对话失败。

你能做什么

  • 控制 git 上下文的信息密度：只取最近 20 条提交 + 当前 diff，不要把整个 git log 塞进上下文
  • 让上下文组装容错：LSP 超时、git 命令失败时降级到空值，不要因为上下文组装失败导致整个对话失败
  • 预先缓存稳定的上下文部分：工具描述在会话开始时计算一次，git 状态在每次用户消息时更新
  • 测量上下文大小：定期检查各部分上下文的 token 占比，过大的部分考虑摘要或截断
