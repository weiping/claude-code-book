# 第 23 章：会话系统——标识、持久化与恢复

> 用户关闭了 Claude Code，第二天重新打开——对话历史还在吗？会话 ID 是怎么生成的？如果两个终端同时运行 Claude Code 处理同一个项目，它们的历史怎么隔离？最麻烦的是：如果中途崩溃，半写入的会话文件会不会导致"历史恢复时数据损坏"？

这是**持久化系统的经典三难**：原子性 vs 性能 vs 可恢复性。Claude Code 用 JSONL 追加写入解决原子性，用内容寻址哈希解决隔离，用重放机制解决崩溃恢复。本章拆解每一层的工程决策。

---
## 23.1 JSONL 追加写入设计
### 定义
会话对话记录以 JSONL 格式存储在 `.claude/projects/{hash}/{sessionId}.jsonl`。每条对话（用户输入 + Claude 回复）序列化为一行 JSON，直接追加到文件末尾。不覆写、不重排。
### JSONL 格式示例
```jsonl
{"type":"user","timestamp":"2026-04-04T10:15:32Z","content":"优化这个函数","tokens":15}
{"type":"assistant","timestamp":"2026-04-04T10:15:45Z","content":"...建议...","tokens":450}
{"type":"user","timestamp":"2026-04-04T10:16:02Z","content":"加错误处理","tokens":8}
{"type":"assistant","timestamp":"2026-04-04T10:16:15Z","content":"...代码...","tokens":320}
```
### 为什么追加而非覆写
**追加写入的优点**：
1. **崩溃安全**：进程在写入中途崩溃，已写入的行不会损坏
2. **减少 I/O**：只写入新增行，不需要读-修改-写整个文件
3. **增量读取**：可以从上次读取位置继续，支持流式处理
4. **原子性简化**：单行 JSON 是原子单位，不会出现"半行"的损坏情况
**对比覆写方案**：
```
覆写方案：
1. 读取整个文件到内存
2. 追加新行
3. 写回整个文件
   ❌ 进程在步骤 3 中途崩溃 → 整个文件丢失
追加方案：
1. 直接追加一行
   ✓ 进程在写入中途崩溃 → 仅当前行可能不完整，前面的行安全
```
**核心权衡**：追加方案牺牲了"随机修改旧对话"的能力，换来了"崩溃安全 + 高效增量"。对于会话记录这个用途是完全可以接受的。

## 23.2 会话标识与路径构造
### 定义
每个会话都有唯一的 sessionId（UUID）。对话文件路径由三部分组成：
```
~/.claude/projects/{projectHash}/{sessionId}.jsonl
其中：
  projectHash = hash(getOriginalCwd())  # 根据初始工作目录生成
  sessionId = uuid()                     # 128-bit 随机 UUID
```
### 设计意图
为什么用项目哈希而不是项目路径？
- 路径很长且可能包含特殊字符
- 哈希（如 sha256）能生成固定长度的安全目录名
- 支持跨文件系统共享（相同项目路径哈希值相同）
**隔离机制**：
- 同一个项目的多个会话 → 不同 sessionId → 不同 JSONL 文件
- 不同项目的会话 → 不同 projectHash → 不同目录
- 跨项目共享会话规则 → 同一个 ~/.claude/config.yaml
### 实际例子
```
项目1：/home/user/projects/app-a
  projectHash = sha256(path) = "a7c3d5f..."
  会话1: ~/.claude/projects/a7c3d5f/uuid-1.jsonl
  会话2: ~/.claude/projects/a7c3d5f/uuid-2.jsonl
项目2：/home/user/projects/app-b
  projectHash = sha256(path) = "b8f2e9k..."
  会话1: ~/.claude/projects/b8f2e9k/uuid-3.jsonl
```

## 23.3 MAX_TRANSCRIPT_READ_BYTES 与大会话处理
### 定义
`src/utils/sessionStorage.ts:229`：`const MAX_TRANSCRIPT_READ_BYTES = 50 * 1024 * 1024` （50MB）
超过 50MB 的会话文件会被截断（只读取最近 50MB），标注为 `truncated: true`。
### 实际数字
```
50MB JSONL 包含：
  - 平均每行 500 bytes（包含元数据、token 计数等）
  - 50MB ÷ 500 bytes = ~100,000 行
  - = 100,000 条对话（用户输入 + Claude 回复 是 2 行）
  - = 50,000 个"轮次"
  - = ~1000 万 tokens
  - = 3-4 个月的日常使用
```
### 权衡分析
50MB 的限制平衡了：
| 方面 | 太小(<1MB) | 50MB | 太大(无限) |
|------|----------|------|---------|
| 保留历史 | 1-2周 | 3-4个月 | 无限 |
| 内存占用 | 1MB | 50MB | 几GB |
| 启动速度 | 50ms | 500ms | 5+ 秒 |
| 单次 API 请求 | 快速 | 正常 | 可能超时 |

## 23.4 /resume 命令与会话恢复
### 定义
`/resume` 命令：
1. 扫描 ~/.claude/projects/ 所有会话文件
2. 按**修改时间**排序，找最近活跃的
3. 读取其 JSONL 文件（受 MAX_TRANSCRIPT_READ_BYTES 限制）
4. 恢复对话历史到新 REPL 会话
### 为什么用修改时间
修改时间反映**活跃度**而非创建时间。用户通常想恢复最近活跃的会话，而非最早创建的。
**对比分析**：
```
使用创建时间：/resume 恢复 2 个月前的 v1.0 项目旧会话 ❌
使用修改时间：/resume 恢复今天早上的活跃会话 ✓
```

## 23.5 SessionKind 与并发隔离
### 定义
`src/utils/concurrentSessions.ts:18`：
```typescript
export type SessionKind = 'interactive' | 'bg' | 'daemon' | 'daemon-worker'
```
四种会话类型对应四种场景：
- **interactive**：终端中的交互式 REPL（用户在等待响应）
- **bg**：后台任务（不占用终端，异步执行）
- **daemon**：守护进程（长时间运行）
- **daemon-worker**：守护进程内的工作线程（并发子任务）
每种有独立的 sessionId 和 JSONL 文件，避免相互污染。
### 并发写入冲突避免
同时开两个终端窗口时：
```
窗口1: sessionId = uuid-1, kind = interactive
  JSONL 路径：~/.claude/projects/{hash}/uuid-1.jsonl
窗口2: sessionId = uuid-2, kind = interactive  
  JSONL 路径：~/.claude/projects/{hash}/uuid-2.jsonl
✓ 不同的文件路径，无写入冲突
✓ 都能独立地追加，不相互干扰
```
**如果共用同一个文件会怎样**：
```
两个窗口同时追加 → 文件锁竞争 → 写入乱序或损坏
```

通过 sessionId 隔离，彻底避免了这个问题。

### 为什么 JSONL 而不是 SQLite 或 JSON？

| 格式 | 写入 | 并发安全 | 崩溃恢复 | 人类可读 |
|------|------|---------|---------|---------|
| JSON（整体覆写）| 快 | ❌（竞态） | ❌（半写） | ✓ |
| SQLite | 中 | ✓（锁）| ✓ | ❌ |
| JSONL（追加）| 快 | ✓（追加原子）| ✓（跳过损坏行）| ✓ |

JSONL 的 `append` 操作是操作系统级别的原子操作——即使两个进程同时追加，每行也是完整的。崩溃时最多丢失最后一行，之前的历史完好无损（`src/utils/sessionStorage.ts:229`）。

### 为什么用项目路径哈希而不是随机 UUID 作为目录名？

随机 UUID 每次都不同，无法通过路径推断"这是哪个项目的历史"。

项目路径哈希（`src/bootstrap/state.ts:331`）让：
- 相同项目的历史总是在同一个目录
- `/resume` 命令不需要搜索所有目录，直接定位
- 磁盘占用可预测（同项目历史聚合，不会无限散列）

代价：哈希碰撞理论上存在，实践中可忽略。


## 模式提炼
1. **Append-only JSONL**：采用追加写入，每条对话一行 JSON，崩溃安全
2. **sessionId + projectHash 隔离**：两层组合确保会话不混淆，支持多项目多会话
3. **MAX_TRANSCRIPT_READ_BYTES 限制**：50MB 限制平衡了历史保留、内存占用、启动速度
4. **按修改时间 /resume**：按活跃度恢复，符合用户直觉
5. **SessionKind 多类型隔离**：四种类型各自独立，避免并发写入冲突

## 模式提炼

### 追加写入日志（Append-Only Log）

**解决的问题**：对话历史的持久化需要同时满足崩溃安全（写入途中崩溃不损坏已有数据）和并发安全（多个进程不会互相覆盖）。

**核心做法**：每条消息序列化为一行 JSON（JSONL 格式），用 `appendFileSync` 追加写入而非覆写整个文件。每次只追加一行，系统崩溃最多丢失最后一条，不会损坏之前的记录。

**前置条件**：接受"无法修改历史"的约束（历史只能追加，不能更新），需要额外的"tombstone"记录来标记删除。

**源码证据**：`src/bootstrap/session/` — JSONL 追加写入是 Claude Code 会话历史的核心存储机制。

### 内容寻址标识（Content-Addressed Identity）

**解决的问题**：用随机 UUID 作为会话 ID 简单，但无法判断两个会话是否基于相同配置，也无法复现历史问题。

**核心做法**：用会话的关键配置（项目路径、用户 ID、模型名等）的哈希作为会话标识。相同配置产生相同 ID，方便调试和对比；不同配置产生不同 ID，天然隔离。

**源码证据**：`.claude/projects/{hash}/` — 目录名是项目路径的哈希，相同项目的会话存在同一目录下。

## 延伸：会话 ID 的状态管理——getSessionId 与 switchSession

会话 ID 不只是一个字符串——它在 `src/bootstrap/state.ts` 中通过专门的 getter/setter 管理，以确保所有模块看到的是一致的当前会话：

```typescript
// src/bootstrap/state.ts（getSessionId 约在第200行）
export function getSessionId(): SessionId {
  return state.sessionId
}

// switchSession 在 /resume 命令执行时调用
export function switchSession(newSessionId: SessionId): void {
  state.sessionId = newSessionId
  // 清除旧会话的文件状态缓存，确保新会话读到最新文件状态
  state.fileStateCache = new Map()
}
```

**为什么不直接用 `state.sessionId`？** 因为 `getSessionId` 是 bootstrap 模块的公共接口，ESLint 规则 `bootstrap-isolation` 会检测模块是否通过接口访问（而非直接访问内部 `state` 对象）。`switchSession` 的存在说明会话切换不只是改 ID，还需要联动清除文件缓存——这是 `/resume` 命令能正确恢复旧会话状态的基础（`src/bootstrap/state.ts`）。

## 踩坑

### ❌ 用覆写（覆盖整个文件）而非追加写入保存会话历史

```typescript
// ❌ 错误：并发写入时数据损坏
fs.writeFileSync(sessionFile, JSON.stringify(allMessages))
```

两个进程（主 REPL 和后台同步）同时写同一个文件，后一个写入会覆盖前一个的内容。

**正确做法**：JSONL 追加写入（`appendFileSync`），每条消息是独立的一行，天然支持并发安全（`src/bootstrap/session`）。

### ❌ 用时间戳或随机数生成会话 ID

时间戳 ID 在相同配置下无法复现（不便于调试），随机 ID 无法判断两个会话是否配置相同。Claude Code 用配置内容的哈希作为会话标识，相同的项目/用户/配置产生相同的 ID（便于缓存和复现）。

### ❌ 崩溃后重载时不处理不完整的 JSONL 行

进程崩溃时可能正在写一行 JSON，磁盘上是半截的 JSON。重载时如果不用 try/catch 包裹每行的解析，遇到损坏行就整个历史加载失败。

## 你能做什么

- **用 JSONL 追加写入实现崩溃安全**：每条消息独立一行，崩溃时最多丢失最后一条，不会损坏整个历史
- **会话 ID 用内容哈希而非随机**：相同配置的会话产生相同 ID，便于调试和 A/B 对比
- **实现崩溃恢复**：JSONL 重载时对每行用 try/catch，遇到损坏行跳过并记录警告，不要因为一行损坏导致整个历史无法加载
- **会话文件按项目哈希分目录存储**：避免所有会话文件堆在同一个目录，大量会话时文件系统性能下降
