# 第 32 章：权限请求的跨进程 Mailbox 机制
> 当一个在子进程中运行的 Teammate 需要执行"删除文件"操作时，它无法直接显示权限对话框给用户（对话框在主窗口中）。系统如何在子进程和主进程之间异步地传递权限请求，然后回传用户的决策？
---
## 32.1 跨进程通信的困难
### 定义
在第 30 章，我们看到了三种 Task 执行模式。其中，**RemoteAgentTask 在完全独立的子进程中运行**。
但独立的子进程有一个核心问题：**无法直接与主进程的 UI 通信**。
```
子进程（Agent B）
  ├─ 需要执行"删除文件"
  ├─ 检查权限规则 → 规则说"需要用户批准"
  ├─ 想要显示权限对话框
  └─ ❌ 但对话框库是主进程的，子进程无法控制它
子进程该怎么办？
  A) 阻塞等待？ → 会导致死锁（主进程等子进程结果，子进程等主进程批准）
  B) 直接拒绝？ → 太严格，限制功能
  C) 异步通信？ → ✅ 这就是 Mailbox 的作用
```
### 设计意图：Mailbox 机制
**Mailbox** 是一个消息队列系统，用于在主进程和子进程之间进行异步通信。
在 `src/utils/mailbox.ts` 第 1-50 行：
```typescript
export type Message = {
  id: string
  source: MessageSource        // 'user' | 'teammate' | 'system' | ...
  content: string
  from?: string               // 发件人（如 "researcher@my-team"）
  color?: string              // UI 颜色
  timestamp: string
}
export class Mailbox {
  private queue: Message[] = []
  private waiters: Waiter[] = []  // 等待特定消息的处理器
  send(msg: Message): void {
    // 如果有人在等这个消息，直接转移到他们
    // 否则放入队列
  }
  poll(fn: (msg: Message) => boolean): Message | undefined {
    // 从队列中提取一条消息
  }
  wait(fn: (msg: Message) => boolean): Promise<Message> {
    // 异步等待一条消息
  }
}
```
**关键设计**：Mailbox 不是一个"消息总线"，而是一个**基于预期的匹配系统**——发件人发消息时，如果有人在等待这种消息，会被直接转移而不进入队列。

## 32.2 权限请求的完整流程
### 流程步骤
当 Sub Agent 需要权限时，会发生什么？
**步骤 1：权限检查失败**
```typescript
// Sub Agent 执行
const result = await executeBash("rm -rf /data")
// 权限检查层
if (permissionMode === 'plan') {
  // 需要用户批准，但无法直接显示对话框
  // → 发送权限请求到 Mailbox
}
```
**步骤 2：构造权限请求消息**
```typescript
// 在 Sub Agent（子进程）中
const permissionRequest: Message = {
  id: generateRequestId(),
  source: 'teammate',
  from: 'researcher@my-team',
  content: JSON.stringify({
    type: 'permission_request',
    action: 'bash_execute',
    command: 'rm -rf /data',
    timestamp: Date.now(),
  }),
  timestamp: new Date().toISOString()
}
// 发送到 Mailbox
mailbox.send(permissionRequest)
```
**步骤 3：主进程接收并显示**
```typescript
// 在主进程中
const permissionMsg = await mailbox.wait(
  msg => msg.source === 'teammate' && msg.content.type === 'permission_request'
)
// Main Agent 的 UI 显示权限对话框
showPermissionDialog({
  from: permissionMsg.from,
  action: permissionMsg.content.action,
  command: permissionMsg.content.command,
})
// 用户做出决策
const userDecision = await getUserInput()  // "approve" | "deny"
```
**步骤 4：回传决策**
```typescript
// 主进程发送决策到 Mailbox
const decisionMessage: Message = {
  id: generateRequestId(),
  source: 'system',
  from: 'main',
  content: JSON.stringify({
    type: 'permission_decision',
    requestId: permissionRequest.id,
    decision: userDecision,
    timestamp: Date.now(),
  })
}
mailbox.send(decisionMessage)
```
**步骤 5：Sub Agent 继续执行**
```typescript
// 回到子进程
const decisionMsg = await mailbox.wait(
  msg => msg.content.requestId === permissionRequest.id
)
if (decisionMsg.content.decision === 'approve') {
  // 执行删除
  await executeBash("rm -rf /data")
} else {
  // 拒绝
  throw new PermissionDeniedError(...)
}
```

## 32.3 两种 Mailbox 实现
### 实现一：内存型 Mailbox（InProcessTeammateTask）
在 `src/utils/mailbox.ts` 中：
```typescript
// 同一进程中，Mailbox 就是内存中的队列
export class Mailbox {
  private queue: Message[] = []
  send(msg: Message): void {
    // 直接放入队列
    this.queue.push(msg)
  }
  wait(fn: (msg: Message) => boolean): Promise<Message> {
    // 立即检查队列
    const existing = this.queue.find(fn)
    if (existing) {
      return Promise.resolve(existing)
    }
    // 否则注册一个"等待者"
    // 下次 send() 时，会检查是否有匹配的等待者
    return new Promise(resolve => {
      this.waiters.push({ fn, resolve })
    })
  }
}
```
**性能**：
- 超快（内存操作，毫秒）
- 无 IPC 开销
- 但需要代码纪律（防止状态污染）
### 实现二：文件型 Mailbox（RemoteAgentTask）
在 `src/utils/teammateMailbox.ts` 中：
```typescript
/**
 * Teammate Mailbox - File-based messaging system
 *
 * Each teammate has an inbox file at:
 * .claude/teams/{team_name}/inboxes/{agent_name}.json
 * 
 * Other teammates can write messages to it.
 */
```
**结构**：
```
.claude/teams/my-team/inboxes/
  ├── researcher.json       # researcher Agent 的收件箱
  │   {
  │     "messages": [
  │       {"id": "msg-1", "type": "permission_request", ...},
  │       {"id": "msg-2", "type": "permission_decision", ...}
  │     ]
  │   }
  │
  ├── coder.json           # coder Agent 的收件箱
  │
  └── .lock               # 防止并发写入
```
**工作原理**：
```typescript
// Sub Agent 发送消息
const inbox = getTeammateInbox('researcher@my-team')
appendMessageToInbox(inbox, permissionRequest)
// Main Agent 定期轮询
while (true) {
  const newMessages = readInbox('researcher@my-team')
  for (const msg of newMessages) {
    handlePermissionRequest(msg)
  }
  await sleep(100)  // 轮询间隔
}
// Main Agent 回复
appendMessageToInbox(
  getTeammateInbox('researcher@my-team'),
  decisionMessage
)
```
**性能**：
- 有磁盘 I/O 延迟（几毫秒到几十毫秒）
- IPC 开销（进程间协调）
- 但支持完全隔离和远程执行

## 32.4 Mailbox 的三个关键特性
### 特性 1：非阻塞异步
Mailbox 的 `wait()` 方法是异步的，**不会阻塞整个进程**：
```typescript
// Main Agent 的主循环不会被卡住
async function mainAgentLoop() {
  while (true) {
    // 处理用户输入
    const userMsg = await getUserInput()  // 不会被权限请求卡住
    // 同时，后台监听权限请求
    const permissionPromise = mailbox.wait(
      msg => msg.source === 'teammate'
    )  // 这是异步的，不会阻塞
    // 处理消息
    await Promise.race([
      processUserMessage(userMsg),
      handlePermissionRequest(permissionPromise)
    ])
  }
}
```
### 特性 2：请求-响应配对
每个权限请求都有唯一的 ID，响应必须包含这个 ID。这防止了响应错配：
```typescript
// 请求
{
  id: "req-123",
  type: "permission_request",
  action: "delete_file"
}
// 响应必须包含相同的 ID
{
  requestId: "req-123",
  decision: "approve"
}
```
**为什么重要？**
```
假设有两个子进程同时发送权限请求：
  Sub A: id=req-123, action=delete (危险)
  Sub B: id=req-124, action=read (安全)
如果没有 ID 配对，响应可能乱序：
  Main 批准了 req-123 (delete)
  但子 B 收到了响应 → 子 B 执行 delete，错误！
有 ID 配对，每个子进程知道自己的响应是哪个。
```
### 特性 3：超时机制
为防止子进程无限等待，Mailbox 支持超时：
```typescript
const decisionMessage = await Promise.race([
  mailbox.wait(msg => msg.requestId === permissionRequest.id),
  sleep(30 * 1000)  // 30 秒超时
])
if (!decisionMessage) {
  // 超时了，用户没有及时回应
  throw new TimeoutError("Permission request timed out")
}
```

## 模式提炼
### 模式一：异步消息队列与等待者模式（Async Queue with Waiter Pattern）
**解决的问题**：在不阻塞主进程的情况下，如何让多个子进程等待主进程的响应？
**核心做法**：
- 消息到达时，先检查是否有"等待者"在等这个消息
- 如果有，直接转移给等待者（不进队列）
- 如果没有，放入队列
- 等待者通过 Promise 异步等待
**前置条件**：需要支持异步/await，且消息必须有清晰的匹配谓词。
**源码证据**：`src/utils/mailbox.ts`

---

### 模式二：文件系统作为 IPC 通道（Filesystem-based IPC）
**解决的问题**：跨进程、跨机器的通信，但又不想依赖复杂的 RPC 框架。
**核心做法**：
- 每个 Agent 有自己的"收件箱"文件
- 其他 Agent 通过写入文件来发送消息
- 使用锁文件防止并发写冲突
- 定期轮询检查新消息
**前置条件**：需要共享的文件系统，且支持原子写入和文件锁。
**源码证据**：`src/utils/teammateMailbox.ts`

---

### 模式三：请求-响应的 ID 配对（Request-Response ID Pairing）
**解决的问题**：多个并发请求时，响应可能乱序。
**核心做法**：
- 每个请求生成唯一 ID
- 响应必须包含请求 ID
- 等待时指定 "等待 ID 为 X 的响应"
**前置条件**：消息结构必须支持 ID 字段。

## 32.6 权限决策的缓存与审计
### 缓存策略
为了避免重复问用户相同的权限，系统缓存决策：
```
首次：Sub Agent 执行 bash → 询问用户 → 获得批准
第二次：Sub Agent 执行相同的 bash → 查询缓存 → 直接批准（无需提问）
缓存键：
  {
    agent: "researcher@my-team",
    action: "bash_execute",
    command: "rm -rf /data"
  }
```
**缓存的好处**：
- 减少用户交互
- 加快 Agent 执行
**缓存的风险**：
- 如果 Agent 被劫持，缓存的权限可能被滥用
### 审计日志
每次权限决策都被记录：
```json
{
  "timestamp": "2026-04-04T21:55:00Z",
  "requestId": "req-123",
  "agent": "researcher@my-team",
  "action": "bash_execute",
  "command": "rm -rf /data",
  "decision": "approve",
  "decidedBy": "user",
  "cacheHit": false
}
```

Mailbox 机制的核心数据结构定义在 `src/utils/swarm/permissionSync.ts:49`：

`generateRequestId`（`src/utils/swarm/permissionSync.ts:160`）确保权限请求的幂等性——如果 Worker Agent 因网络问题重发同一个权限请求，Leader 能识别出是重复请求，不会重复弹出对话框。

`leaderPermissionBridge`（`src/utils/swarm/leaderPermissionBridge.ts`）解决了一个微妙的问题：同进程 Worker（`InProcessTeammateTask`）无法通过文件系统 Mailbox 通信，但需要使用 Leader 的权限对话框。Bridge 模式让 Leader 的 React 状态（`setToolUseConfirmQueue`）对非 React 代码可访问：

```typescript
// src/utils/swarm/leaderPermissionBridge.ts:30
export function registerLeaderToolUseConfirmQueue(
  setter: SetToolUseConfirmQueueFn
): void {
  registeredSetter = setter  // REPL 注册自己的 setState
}
// 同进程 Worker 调用此函数，效果等同于 Leader 弹出权限对话框
```

这是 Claude Code 少数几个需要打破 React 单向数据流的地方——跨进程的权限同步逼得系统设计绕开了 React 的状态管理。

**Mailbox 目录结构**：`getPermissionDir(teamName)`（`src/utils/swarm/permissionSync.ts:112`）返回类似 `~/.claude/swarm/{teamName}/permissions/` 的路径。每个待处理的权限请求在这个目录下创建一个 JSON 文件，文件名是 `generateRequestId()`（`src/utils/swarm/permissionSync.ts:160`）生成的唯一 ID。Leader 轮询这个目录，发现新文件就弹出对话框；用户响应后，Leader 将结果写入对应的 response 文件。这是文件系统作为异步消息队列的经典用法——简单、可靠、崩溃安全。

## 踩坑

### ❌ 权限请求的 reply 没有超时机制，发送方永久等待

```typescript
// ❌ 错误：如果接收方 crash，这个 Promise 永远不会 resolve
const approval = await mailbox.request(PermissionRequest)
```

用户关闭了权限对话框、接收方进程 crash——任何情况都可能导致 reply 永不到达。权限请求应该有超时（比如 30 秒），超时后自动拒绝（`src/utils/mailbox.ts`）。

### ❌ 用共享内存替代 Mailbox 进行跨进程状态共享

Node.js 进程间通过 `SharedArrayBuffer` 共享内存，看起来很高效，但进程 crash 后共享内存的状态是不确定的，另一个进程可能读到损坏的数据。Mailbox 的消息传递模型保证了进程隔离。

### ❌ 把权限请求和普通消息放在同一个 Mailbox 队列里

如果队列里有 100 个普通消息待处理，权限请求要排在最后。但权限请求通常有时效性（用户在等待），需要单独的高优先级通道或超时机制。

## 你能做什么

- **给所有 Mailbox 请求加超时**：权限请求 30 秒无响应自动拒绝，不要让 UI 永久卡在等待状态
- **用 Mailbox 代替共享内存做跨进程通信**：消息传递保证进程隔离，一个进程 crash 不会损坏另一个进程的状态
- **为权限请求设置优先级队列**：权限请求有时效性，应该优先于普通消息被处理
- **在 Mailbox 消息里包含 traceId**：请求和响应的关联通过 traceId 追踪，便于调试异步流程
