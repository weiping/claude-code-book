# 第 31 章：子智能体的隐式继承与记忆隔离
> 当一个 Agent 调用 fork 命令启动一个子 Agent 时，子 Agent 是从零开始，还是自动继承父 Agent 的对话历史和权限设置？这个"继承"的边界在哪里？
---
## 31.1 隐式继承机制：从零到继承
### 定义与困难
在第 30 章，我们看到了 RemoteAgentTask 和 InProcessTeammateTask 都支持"启动一个 Teammate"。但有一个问题：
**新 Agent 从零开始的问题**：
```
情景：Main Agent 经过 10 轮对话，积累了大量上下文
Main Agent 决定启动 Sub Agent 来做一个辅助任务
问题 1：Sub Agent 应该知道前面的 10 轮吗？
  如果不知道 → Sub Agent 的决策基于不完整信息，可能出错
  如果知道   → 如何高效地传递这 10 轮的上下文？
问题 2：Sub Agent 应该继承权限规则吗？
  如果不继承 → 用户需要重复批准相同操作
  如果继承   → 需要防止子 Agent 滥用权限
问题 3：如何区分"继承"和"独立"？
  有时需要 Sub Agent 的完全隔离决策
  有时需要 Sub Agent 遵循 Main 的策略
```
### 设计意图：隐式继承
Claude Code 的解决方案是**隐式继承（Implicit Inheritance）**。
在 `src/tools/AgentTool/forkSubagent.ts` 第 26-50 行：
```typescript
/**
 * Fork subagent feature gate.
 *
 * When enabled:
 * - `subagent_type` becomes optional on the Agent tool schema
 * - Omitting `subagent_type` triggers an implicit fork: the child inherits
 *   the parent's full conversation context and system prompt
 * - All agent spawns run in the background (async) for a unified
 *   `<task-notification>` interaction model
 */
export function isForkSubagentEnabled(): boolean {
  // Feature gate 检查
  // 如果启用，子 Agent 可以自动继承父的上下文
}
```
**关键词**："Omitting `subagent_type` triggers an implicit fork"——不指定类型时自动触发隐式 fork。
### 继承的具体内容
当 Sub Agent 隐式继承时，继承了什么？
| 内容 | 是否继承 | 方式 |
|------|--------|------|
| **对话历史** | ✅ 是 | 完整复制前 N 轮消息 |
| **系统提示** | ✅ 是 | 继承相同的 system prompt |
| **工具集** | ✅ 是 | `tools: ['*']`，继承父的所有工具 |
| **权限规则** | ✅ 是 | 继承 permissionMode（Plan/Auto/YOLO） |
| **API 缓存前缀** | ✅ 是 | 使用父相同的 cache_control prefix |
| **上下文变量** | ✅ 是 | 继承 cwd、git status 等 |
| **对话 ID** | ❌ 否 | Sub Agent 有自己的 UUID |
**为什么 API 缓存也继承？**
```
不继承的风险：
  Parent 的对话前 10 轮设置了 Prompt Cache（缓存成本 3 块钱）
  Sub Agent 用完全不同的 cache prefix
  → 无法利用 Parent 的缓存
  → API 成本翻倍
继承的优势：
  Sub Agent 使用父相同的 cache prefix
  → 可以重用 Prompt Cache
  → 节省成本
```

## 31.2 fork 的工作流
### 工作流步骤
当用户输入 `/fork <directive>` 时，系统如何处理？
**步骤 1：解析 Fork 指令**
在 `src/tools/AgentTool/forkSubagent.ts` 中，fork 指令被解析为：
```typescript
const forkDirective = {
  context: "parent_context",      // 父 Agent 的完整对话
  systemPrompt: "...",             // 继承的系统提示
  tools: ['*'],                    // 所有工具
  agentType: 'fork',               // 合成类型（用于分析）
}
```
**步骤 2：创建记忆快照**
调用 `agentMemorySnapshot.ts` 中的函数，为 Sub Agent 创建初始记忆：
```typescript
const snapshot = {
  parentMessages: [
    {role: "user", content: "..."},
    {role: "assistant", content: "..."},
    // ... 前 10 轮
  ],
  timestamp: Date.now(),
  forkId: "fork-xyz"  // 追踪来源
}
```
**步骤 3：启动 Sub Agent Task**
选择合适的 Task 容器（通常是 InProcessTeammateTask），执行 `runAgent()`：
```typescript
// 伪代码
spawnMultiAgent([{
  agentId: 'fork-' + randomUUID(),
  agentName: 'fork_assistant',
  teamName: 'main_swarm',
  initialContext: snapshot,
  inheritedPermissionMode: parent.permissionMode,
}])
```
**步骤 4：Sub Agent 执行**
Sub Agent 运行 `query.ts` 的核心循环，但：
- 起始消息历史中已包含父的前 N 轮
- 系统提示相同
- 权限模式继承
**步骤 5：结果回传**
Sub Agent 完成后，返回结果消息到 Main Agent：
```typescript
{
  role: "assistant",
  content: [{
    type: "tool_result",
    from: "fork-xyz",
    result: "Sub Agent 的最终输出"
  }]
}
```

## 31.3 记忆隔离：agentMemorySnapshot
### 定义
**agentMemorySnapshot** 是 Sub Agent 的"记忆拍照"机制。当 Sub Agent 执行时，它的对话历史被隔离存储，不直接污染 Main Agent 的历史。
在 `src/tools/AgentTool/agentMemorySnapshot.ts` 第 1-30 行：
```typescript
/**
 * Agent memory snapshots
 * 
 * Each agent (or subagent fork) has its own memory snapshot directory.
 * When a subagent completes, its memory is captured and can be:
 * 1. Reviewed independently
 * 2. Synced back to parent if needed
 * 3. Reused for similar tasks
 */
```
### 记忆隔离的三个层面
#### 层面 1：文件系统隔离
记忆快照存储在不同的目录：
```
.claude/
├── agent-memory/
│   ├── parent_agent/
│   │   ├── chat_history.json
│   │   └── memory.json
│   │
│   └── research_agent/
│       ├── chat_history.json
│       └── memory.json
│
└── agent-memory-snapshots/
    ├── fork_assistant/
    │   ├── snapshot.json
    │   └── .snapshot-synced.json
    │
    └── verification_agent/
        ├── snapshot.json
        └── .snapshot-synced.json
```
**为什么分开存储？**
```
好处：
  ✓ 每个 Agent 的记忆完全独立
  ✓ 便于调试（可单独查看某个 Agent 的思考过程）
  ✓ 便于重用（同一个 Agent 的多次执行可对比）
  ✓ 便于删除（某个 Agent 出错，只需删除其快照）
成本：
  ✗ 磁盘占用增加
  ✗ 同步成本（若要合并记忆）
```
#### 层面 2：内存隔离
在运行时，Sub Agent 的消息历史被包装成特殊的消息类型：
```typescript
// Main Agent 的消息历史
[
  {role: "user", content: "启动研究 Agent"},
  {role: "assistant", content: "我启动了..."},
  {
    role: "tool",
    type: "agent_result",    // 特殊类型
    agent_id: "researcher",
    messages: [               // Sub Agent 的完整历史
      {role: "user", content: "研究主题"},
      {role: "assistant", content: "我发现..."},
    ]
  }
]
```
**为什么用特殊消息类型？**
```
优点：
  ✓ Main Agent 可以分辨"自己的对话"和"Sub Agent 的对话"
  ✓ 便于 UI 展示（可折叠 Sub Agent 的对话）
  ✓ 便于权限审计（知道哪个操作来自哪个 Agent）
缺点：
  ✗ API 调用时需要"翻译"成标准消息格式
  ✗ 如果 Sub Agent 很多，消息树会很深
```
#### 层面 3：权限隔离
每个 Agent 有独立的权限决策上下文：
```typescript
// Sub Agent 的权限决策
{
  permissionMode: "auto",           // 可能与 Main 不同
  approvedActions: [
    "bash_read",
    "file_read"
  ],
  deniedActions: [
    "file_delete"                   // 即使 Main 允许，Sub 也可以拒绝
  ]
}
```

## 31.4 隐式继承的两个陷阱与解决
### 陷阱 1：无限递归（无限 fork）
**问题**：如果 Sub Agent A fork 了 Sub Agent B，Sub Agent B 又 fork 了 Sub Agent C，能否无限继续？
```
Main (级别 0)
  → Fork A (级别 1)
      → Fork B (级别 2)
          → Fork C (级别 3)
              → ... 无限继续？
```
**风险**：
- 消息树无限深化
- API token 消耗失控
- 调试变得不可能
**解决方案**：限制 fork 深度
在 `src/tools/AgentTool/forkSubagent.ts` 中有深度检查：
```typescript
const MAX_FORK_DEPTH = 5  // 最多 fork 5 层
function checkForkDepth(currentAgent: Agent): boolean {
  if (currentAgent.forkDepth >= MAX_FORK_DEPTH) {
    throw new Error("Fork depth exceeded")
  }
  return true
}
```
### 陷阱 2：权限提升风险
**问题**：如果 Main Agent 有"删除文件"权限，Sub Agent 继承这个权限，会怎样？
```
Main Agent 的权限：
  - 允许删除文件（用户已批准）
Sub Agent 继承后：
  - Sub Agent 也可以删除文件
  - 但用户没有显式批准 Sub Agent 执行此操作
  - 风险：Sub Agent 执行未预期的删除
```
**风险分级**：
```
低风险操作（如 read_file）：
  → 继承是安全的
中风险操作（如 create_file）：
  → 继承但需要 Sub Agent 显式确认
高风险操作（如 delete_file）：
  → 继承但每次都需要重新授权
```
**解决方案**：权限模式分化
```typescript
// Main Agent 可能是 YOLO（完全信任）
// 但 Sub Agent 可能被限制为 Plan（需要计划批准）
const inheritedPermissionMode = Math.min(
  parent.permissionMode,      // 不能比父更宽松
  'auto'                      // 但子可以选择更严格
)
```

## 设计决策：为什么这样设计子 Agent？

### 为什么 forkSubagent 用"隐式继承"而不是显式参数传递？

显式传递方案：
```typescript
// 显式方案（过于繁琐）
await runAgent({
  model: parent.model,
  permissions: parent.permissions,
  workingDir: parent.workingDir,
  systemPrompt: parent.systemPrompt,
  // ...20个参数
})
```

Claude Code 的隐式继承方案：子 Agent 通过 `forkSubagent()` 从父 Agent 的上下文派生，自动继承模型设置、权限规则、工作目录等。只需显式传递**差异**（任务描述、额外工具）而非全量配置（`src/tools/AgentTool/runAgent.ts:248`）。

**权衡**：隐式继承的风险是子 Agent 可能继承不应有的权限。`agentMemorySnapshot` 的隔离机制解决了这个问题：每个子 Agent 有独立的内存快照，即使使用父 Agent 的配置，也不能访问父 Agent 的内存历史（`src/tools/AgentTool/runAgent.ts`）。

### 为什么 runAgent 是 async generator 而不是普通 async 函数？

```typescript
// src/tools/AgentTool/runAgent.ts:248
export async function* runAgent({...}: RunAgentParams): AsyncGenerator<AgentEvent>
```

普通 async 函数只能在最后 return 一次。子 Agent 的执行是一个多步骤的过程（每轮 LLM 调用都有中间事件），调用方（Swarm 或 AgentTool）需要实时看到进度更新。

`async function*` 让 `runAgent` 可以在执行过程中 `yield` 事件（工具调用开始、工具结果、进度更新），调用方通过 `for await...of` 逐事件处理，与 `query.ts` 同构——子 Agent 的对话循环和主循环是同一种抽象。

### 为什么 resumeAgent 需要重建消息历史而不是直接恢复？

直接恢复方案：从磁盘读取完整的消息列表，传给 `runAgent` 继续执行。

实际问题：`tool_use` 和 `tool_result` 必须成对出现，崩溃可能导致最后一个 `tool_use` 没有对应的 `tool_result`。直接恢复会导致 API 报错（`src/tools/AgentTool/resumeAgent.ts:42`）。

`readAgentMetadata` + `reconstructMessages` 的重建步骤：过滤不完整的 tool_use 对，确保恢复后的消息历史对 Claude API 是合法的。这是"崩溃安全恢复"的核心。


## 模式提炼
### 模式一：隐式继承与显式隔离（Implicit Inheritance with Explicit Isolation Points）
**解决的问题**：如何让子 Agent 自动继承大部分状态，同时在关键点上保持隔离？
**核心做法**：
- 通过特殊消息类型（`agent_result`）包装 Sub Agent 的结果
- 记忆存储在独立目录
- 权限决策有独立的上下文
**前置条件**：需要定义"哪些状态可继承""哪些必须隔离"。
**源码证据**：`src/tools/AgentTool/forkSubagent.ts`、`src/tools/AgentTool/agentMemorySnapshot.ts`

---

### 模式二：深度限制与递归防护（Depth Limiting for Recursive Structures）
**解决的问题**：无限递归可能导致成本失控。
**核心做法**：在 fork 时检查当前深度，超过阈值则拒绝。
**前置条件**：每个 Agent 记录自己的 fork 深度。
**源码证据**：MAX_FORK_DEPTH 检查

---

### 模式三：权限继承的最小权限原则（Principle of Least Privilege in Inheritance）
**解决的问题**：Sub Agent 不应该自动获得比 Main 更高的权限。
**核心做法**：`inheritedPermissionMode = min(parent.permissionMode, 'auto')`
**前置条件**：定义权限模式的偏序关系（YOLO > Auto > Plan）。

## 31.6 与 runAgent 的同构性
### 定义
从某个角度看，**fork 本质上就是调用 runAgent()，但起始消息历史不同**。
```typescript
// 普通 Sub Agent
runAgent({
  agentType: 'researcher',
  initialMessages: [],            // 从空开始
  systemPrompt: getSystemPrompt()
})
// Fork Sub Agent
runAgent({
  agentType: 'fork',
  initialMessages: parentMessages, // 继承父的历史
  systemPrompt: getSystemPrompt()  // 相同的系统提示
})
```
**同构性的好处**：
- 不需要两套不同的执行引擎
- Bug 修复自动应用到两种场景
- 降低代码复杂性

## 延伸：runAgent 函数签名与 resumeAgent 恢复路径

`runAgent` 是子 Agent 执行层的核心（`src/tools/AgentTool/runAgent.ts:248`）：

```typescript
// src/tools/AgentTool/runAgent.ts:248
export async function* runAgent({
  agent,
  initialMessages,
  tools,
  abortSignal,
  parentContext,
  // ...更多参数
}: RunAgentParams): AsyncGenerator<AgentEvent> {
  // 这是一个 async generator，与 query.ts 同构
  // 子 Agent 的整个对话循环在这里展开
}
```

这个签名的关键点：**`runAgent` 也是 async generator**，与主循环的 `query.ts` 同构——子 Agent 的对话是一个完整的、独立的生成器循环，不是单次函数调用。

`resumeAgent`（`src/tools/AgentTool/resumeAgent.ts:42`）处理 `/resume` 后的子 Agent 状态重建：

```typescript
// src/tools/AgentTool/resumeAgent.ts:42
export async function resumeAgentBackground({
  agentId, sessionPath
}: ResumeParams): Promise<ResumeAgentResult> {
  const metadata = await readAgentMetadata(asAgentId(agentId))
  // 从快照重建 Agent 的内存状态
  return reconstructMessages(metadata)
}
```

`readAgentMetadata` 读取磁盘上的 Agent 快照（`agentMemorySnapshot.ts`），`reconstructMessages` 从快照中重建消息历史，让子 Agent 从中断点继续（而非重新开始）。这是多 Agent 系统中"崩溃恢复"的关键路径。

## 踩坑

### ❌ 子 Agent 继承父 Agent 的所有权限，违反最小权限原则

父 Agent 有写入 `/project` 的权限，不代表"生成代码摘要"的子 Agent 也需要写权限。继承所有权限意味着子 Agent 的任意漏洞（提示注入、恶意工具）都能被利用来执行不该有的操作（`src/agents/agentContext.ts`）。

### ❌ 通过子 Agent 的内存传递父 Agent 的 API 凭证

```typescript
// ❌ 错误：凭证传入子 Agent 的上下文
await subAgent.run({ context: { apiKey: process.env.SECRET_KEY } })
```

子 Agent 的执行上下文可能被其他工具读取或者通过日志泄漏。凭证应该在子 Agent 真正需要时临时注入，而不是放在传递的上下文里。

### ❌ 子 Agent 任务完成后不清理创建的临时资源

子 Agent 可能创建了临时文件、设置了环境变量、启动了子进程。如果没有清理注册（`process.on('exit', cleanup)`），这些资源在子 Agent 结束后继续占用系统，累积后导致资源耗尽。

## 你能做什么

- **为子 Agent 计算"所需权限集合"，而非继承所有权限**：父 Agent 权限 ∩ 任务所需权限 = 子 Agent 权限
- **子 Agent 完成后主动触发清理**：临时文件、环境变量、子进程都应该在任务结束时清理
- **在子 Agent 的上下文里不包含父 Agent 的凭证**：凭证在需要时通过安全通道临时注入，不放在传递的上下文对象里
- **记录子 Agent 的父子关系**：日志里的 `parentAgentId` 字段让你能重建完整的任务树，便于调试复杂的多层 Agent 系统
