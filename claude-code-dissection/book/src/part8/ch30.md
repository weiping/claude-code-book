# 第 30 章：Task 的三种执行模式
> 当 Swarm 决定启动一个 Teammate 时，系统怎么知道该在"同一个进程"运行它，还是"启动一个独立的子进程"？这个选择对性能、隔离性和功能有什么影响？
---
## 30.1 Task 接口的核心设计
### 定义
在第 29 章，我们看到 Task 是 Teammate 的"执行容器"。但什么是 Task？
**Task 接口**（在 `src/Task.ts` 中定义）：
```typescript
export interface Task {
  name: string                       // 任务名称，如 "InProcessTeammateTask"
  type: string                       // 任务类型标识
  kill(taskId: string, setAppState): void  // 杀死任务的方法
}
```
看起来很简洁，但它隐藏了巨大的差异。当调用 `kill()` 时：
- **InProcessTeammateTask**：只需切换状态，任务立即停止
- **RemoteAgentTask**：需要发送 IPC 消息给子进程，等待确认
- **LocalAgentTask**：触发 AbortController，中断执行栈
同样的接口，三种完全不同的实现。
### 为什么需要三种？
```
假设只有 InProcessTeammateTask（同进程）：
  优点：快速，无 IPC 开销
  缺点：
    - 两个 Agent 的全局状态会互相污染
    - 无法在不同机器上运行
    - 权限检查很难真正隔离
假设只有 RemoteAgentTask（子进程）：
  优点：完全隔离，可以远程
  缺点：
    - 每个任务都要启动一个进程，开销大
    - 简单的 Teammate 不值得这个开销
    - 网络通信的延迟
设计选择（三种并存）：
  ✓ 快速场景用 InProcessTeammateTask
  ✓ 高隔离需求用 RemoteAgentTask
  ✓ 轻量级用 LocalAgentTask
```
---
## 30.2 InProcessTeammateTask：同进程执行
### 定义与设计
**InProcessTeammateTask** 让 Teammate 和 Main Agent 运行在同一个 Node.js 进程中。但如何避免状态污染？
答案：**AsyncLocalStorage**（Node.js 18+ 提供的 Context API）。
在 `src/tasks/InProcessTeammateTask/InProcessTeammateTask.tsx` 第 1-20 行：
```typescript
/**
 * InProcessTeammateTask - Manages in-process teammate lifecycle
 *
 * This component implements the Task interface for in-process teammates.
 * Unlike LocalAgentTask (background agents), in-process teammates:
 * 1. Run in the same Node.js process using AsyncLocalStorage for isolation
 * 2. Have team-aware identity (agentName@teamName)
 * 3. Support plan mode approval flow
 * 4. Can be idle (waiting for work) or active (processing)
 */
```
关键词是"AsyncLocalStorage for isolation"——通过异步局部存储创建虚拟隔离。
### 隔离的实现
**AsyncLocalStorage 工作原理**：
```typescript
// 在 Main Agent 的上下文中
const mainContext = new AsyncLocalStorage()
mainContext.run(mainState, () => {
  // Main Agent 的逻辑
  state.get()  // 返回 mainState
  // 启动 Teammate
  const teammateContext = new AsyncLocalStorage()
  teammateContext.run(teammateState, () => {
    // Teammate 的逻辑
    state.get()  // 返回 teammateState，不是 mainState！
  })
})
```
**效果**：
- 同一个全局变量 `state`
- 根据不同的异步执行上下文，返回不同的值
- 无需修改代码，隔离自动发生
### InProcessTeammateTask 的生命周期
在 `src/tasks/InProcessTeammateTask/types.ts` 第 6-35 行：
```typescript
export type InProcessTeammateTaskState = TaskStateBase & {
  type: 'in_process_teammate'
  // 身份信息
  identity: TeammateIdentity
  // 执行状态
  prompt: string
  model?: string                      // 可选的模型覆盖
  selectedAgent?: AgentDefinition
  abortController?: AbortController   // 用于中止任务
  // Plan 模式
  awaitingPlanApproval: boolean
  // 权限模式
  permissionMode: PermissionMode
  // 结果
  error?: string
  result?: AgentToolResult
}
```
**关键字段含义**：
- `abortController`：可以通过调用 `abort()` 来立即停止任务
- `awaitingPlanApproval`：标记是否在等待用户批准计划
- `permissionMode`：每个 Teammate 有独立的权限模式（Plan / Auto / YOLO），互不影响
### 为什么不是所有 Teammate 都用 InProcessTeammateTask？
**开销分析**：
```
情景 1：两个 Teammate 都是 InProcessTeammateTask
  成本：
    ✓ 创建 AsyncLocalStorage 的开销小（毫秒级）
    ✓ 无 IPC 开销
    ✓ 共享内存，快速数据交换
  风险：
    ✗ 如果有 bug 导致状态污染，很难调试
    ✗ 权限检查必须足够严密
情景 2：其中一个是 RemoteAgentTask
  成本：
    ✓ 完全隔离，无法污染彼此
    ✗ 启动子进程（~100-200ms）
    ✗ IPC 通信延迟（毫秒级，但累积）
  适用：高隔离需求（如运行不信任的代码）
```

## 30.3 RemoteAgentTask：子进程执行
### 定义与设计
**RemoteAgentTask** 在一个完全独立的子进程中运行 Teammate。
在 `src/tasks/RemoteAgentTask/RemoteAgentTask.tsx` 第 1-20 行：
```typescript
// RemoteAgentTask 管理子进程生命周期
// 通过 IPC 通道与子进程通信
// 支持在不同机器上运行（基于 teleport/API）
```
### 隔离的实现
**进程隔离 vs AsyncLocalStorage 隔离**：
```
AsyncLocalStorage（InProcess）：
  同一进程的不同异步上下文
  隔离性：中等（依赖代码纪律）
  性能：极快
进程隔离（Remote）：
  完全不同的 Node.js 进程
  隔离性：极强（操作系统级别）
  性能：有 IPC 开销
内存隔离的对比：
  InProcess：
    Main 内存：1GB
    Teammate 1：共享 Main 的 1GB
    Teammate 2：共享 Main 的 1GB
    ───────────
    总计：仍是 1GB（共享）
  Remote：
    Main 内存：1GB
    Teammate 1 进程：500MB
    Teammate 2 进程：500MB
    ───────────
    总计：2GB（独立）
```
### RemoteAgentTask 的通信机制
RemoteAgentTask 通过 IPC 通道与子进程通信。关键的消息类型：
| 消息类型 | 发送者 | 接收者 | 目的 |
|---------|--------|--------|------|
| `task_start` | Parent | Child | 启动子任务 |
| `permission_request` | Child | Parent | 申请权限 |
| `permission_decision` | Parent | Child | 传递权限决策 |
| `task_output` | Child | Parent | 发送任务输出 |
| `task_complete` | Child | Parent | 任务完成 |
| `task_abort` | Parent | Child | 中止任务 |
**流程图**：
```
Parent (Main Agent)
  │
  ├─ 发送 task_start
  │   └─ Child process 启动
  │       └─ Child 执行 runAgent()
  │           ├─ 需要权限 → 发送 permission_request
  │           └─ Parent 收到 → 显示对话框 → 发送 permission_decision
  │           ├─ Child 继续执行
  │           ├─ 生成输出 → 发送 task_output
  │           └─ 完成 → 发送 task_complete
  │
  └─ Parent 收集所有输出，合并结果
```

## 30.4 LocalAgentTask：轻量级隔离
### 定义与设计
**LocalAgentTask** 是一个轻量级的后台任务容器，用于在主进程中运行简单的 Agent。
在 `src/tasks/LocalAgentTask/LocalAgentTask.tsx` 第 1-30 行：
```typescript
/**
 * LocalAgentTask
 * 
 * Background agents that run in the same process as the main session.
 * Used for lower-level agents that don't need full team identity or plan mode.
 */
```
**特点**：
- 不是"Teammate"（没有 agentName@teamName 的身份）
- 运行在后台，不展示在 UI 中
- 简化的生命周期（启动、执行、停止）
### LocalAgentTask 与 InProcessTeammateTask 的区别
| 特性 | InProcessTeammateTask | LocalAgentTask |
|------|----------------------|----------------|
| **身份** | 有（researcher@my-team） | 无 |
| **UI 展示** | 是（独立窗格） | 否（后台） |
| **权限模式** | 有（Plan/Auto/YOLO） | 无 |
| **Plan 模式** | 支持 | 不支持 |
| **隔离方式** | AsyncLocalStorage | AbortController |
| **使用场景** | 多智能体协作 | 后台任务、验证 Agent |
### LocalAgentTask 的生命周期
```typescript
// 伪代码
const localAgentTask = {
  state: {
    type: 'local_agent',
    status: 'running',
    abortController: new AbortController(),
    progress: { currentStep: 1, totalSteps: 5 }
  },
  async execute() {
    try {
      while (!this.state.abortController.signal.aborted) {
        // 执行一个工作单元
        await doWork()
      }
    } catch (err) {
      if (err instanceof AbortError) {
        this.state.status = 'aborted'
      } else {
        this.state.status = 'error'
      }
    }
  },
  kill() {
    this.state.abortController.abort()
  }
}
```

## 30.5 三种 Task 的选择标准
### 决策树
```
何时选择哪种 Task？
【问题 1】需要团队身份和 UI 展示吗？
  ├─ YES → 必须是 InProcessTeammateTask 或 RemoteAgentTask
  │
  └─ NO  → 可能是 LocalAgentTask
       │
       【问题 2】需要完全隔离吗？
       ├─ YES（运行不信任代码、需要远程执行）
       │  └─ RemoteAgentTask ← 选择这个
       │
       └─ NO（快速、简单）
          └─ InProcessTeammateTask ← 选择这个（回到 YES 分支）
               │
               【问题 3】多少个 Teammate 同时运行？
               ├─ 1-2 个 → InProcessTeammateTask（快速）
               │
               └─ 3+ 个 → 考虑混合：部分 InProcess（关键）+ 部分 Remote（非关键）
```
### 成本对比表
| Task 类型 | 启动延迟 | 内存占用 | IPC 开销 | 隔离性 | 合适场景 |
|---------|---------|---------|---------|--------|---------|
| **InProcessTeammateTask** | < 1ms | 共享 | 无 | 中 | 2-3 个协作 Agent |
| **RemoteAgentTask** | 100-200ms | 独立 | 有 | 强 | 高隔离、多并发 |
| **LocalAgentTask** | < 1ms | 共享 | 无 | 低 | 后台验证、辅助任务 |

## 模式提炼
### 模式一：虚拟隔离与物理隔离的权衡（Virtual vs Physical Isolation）
**解决的问题**：如何在性能和隔离性之间平衡？
**核心做法**：
- 虚拟隔离（AsyncLocalStorage）用于快速、相互信任的 Agent
- 物理隔离（子进程）用于高安全性需求
**前置条件**：支持多种 Task 类型、统一的 Task 接口。
**源码证据**：`src/tasks/`、`src/Task.ts`

---

### 模式二：上下文注入与隐式继承（Context Injection Pattern）
**解决的问题**：如何让 Teammate 使用主 Agent 的部分状态（如全局权限规则），同时隔离其他状态（如对话历史）？
**核心做法**：通过 AsyncLocalStorage 的"嵌套"特性，子任务可以访问父任务的某些上下文。
```typescript
// 父任务设置全局规则
globalRulesContext.run({ rules: [...] }, () => {
  // 子任务可以读取规则，但有自己的局部状态
  const rules = globalRulesContext.getStore()  // 继承父的规则
  const localState = localContext.getStore()   // 隔离自己的状态
})
```
**前置条件**：需要清晰的"全局状态"vs"局部状态"划分。
**源码证据**：`src/tasks/InProcessTeammateTask/`

---

### 模式三：统一接口的多实现（Strategy Pattern for Execution）
**解决的问题**：不同的执行模式需要不同的实现，但调用者不应该关心细节。
**核心做法**：所有 Task 实现统一的 Task 接口（name、type、kill）。调用者通过接口调用，而不是直接依赖具体实现。
**前置条件**：需要抽象的 Task 接口。
**源码证据**：`src/Task.ts`、`src/tasks/*/`

---

## 延伸：LocalShellTask 的前台化机制

`LocalShellTask` 是三种 Task 中最特殊的——它支持把后台任务"带到前台"：

```typescript
// src/tasks/LocalShellTask/LocalShellTask.tsx（backgroundExistingForeground 相关）
export class LocalShellTask {
  private isForeground: boolean = false
  
  // 将后台运行的 Shell Agent 切换为前台（用户可以直接交互）
  async bringToForeground(): Promise<void> {
    this.isForeground = true
    // 触发 REPL 切换视图，让用户直接与该 Agent 对话
  }
}
```

**为什么需要前台化？** 在 Plan V2 模式中，多个 Agent 并行工作。用户可能想要"关注"某个特定 Agent 的工作进度，而不是通过主线程的摘要间接了解。`bringToForeground` 让用户能临时"进入"一个 Agent 的视角，直接观察和交互——这是 `LocalShellTask`（独立进程 Agent）特有的能力，`InProcessTeammateTask`（同进程 Agent）没有这个能力，因为进程隔离让直接交互复杂度太高（`src/tasks/LocalShellTask/LocalShellTask.tsx`）。

## 踩坑

### ❌ 用同一种执行模式处理所有任务，忽视任务的特性

I/O 密集型任务（读取多个文件）和 CPU 密集型任务（代码分析）的最优执行模式不同。前者适合异步并发，后者适合 worker thread 或单独进程，混用会导致事件循环阻塞（`src/tasks/`）。

### ❌ 后台任务 crash 时没有通知主进程

```typescript
// ❌ 错误：spawn 后不监听 exit 事件
const child = spawn('node', ['worker.js'])
// 如果 worker.js crash，主进程不知道
```

应该监听 `child.on('exit', (code) => { if (code !== 0) notifyMainProcess() })`，确保子任务的失败不被静默吞掉。

### ❌ 并行任务的日志输出混在同一个流里，无法调试

多个子 Agent 同时输出日志，日志行的顺序混乱，无法追踪某个 Agent 的完整执行流程。每个任务应该有独立的日志流或者每行日志前缀任务 ID。

## 你能做什么

- **根据任务特性选择执行模式**：I/O 密集型用 async 并发，CPU 密集型用 worker thread，避免事件循环阻塞
- **为所有后台任务注册退出清理函数**：`process.on('exit', cleanup)` 确保临时文件、子进程在主进程退出时被清理
- **给每个任务独立的日志前缀**：`[task-abc123]` 前缀让多任务并发时的日志可以按任务过滤
- **监控任务队列长度**：队列持续增长说明消费速度跟不上生产速度，应该增加消费者或限制生产速率
