Claude Code 工程模式

第 25 章：四种执行器——command/prompt/agent/http 的多态设计

"好的扩展点是：你不需要修改主逻辑，就能增加新的行为。"

在同一个 Hook 触发点，四种完全不同的行为可以同时发生：执行一段 bash 命令、向 AI 发起单轮查询、启动一个完整的多轮 Agent、向远程服务发送 HTTP 请求。这四种行为的复杂度差异悬殊——从毫秒级的 shell 调用，到可能耗时 60 秒的 Agent 多轮对话——却统一挂载在同一个 Hook 接口下。

这是多态执行器（Polymorphic Executor）模式的典型实现：四种类型（command/prompt/agent/http）各有独立的 exec*Hook 执行函数，主调度逻辑 executeHooks 只做类型分发，新增执行器类型不需要改动已有的分发代码。

读完本章，你将理解这四种执行器的职责边界、各自的安全策略，以及为什么把"什么都能做"的执行器设计成彼此完全隔离的多态变体，是维护扩展系统长期健康的关键决策。

问题：主调度逻辑不应随执行器类型增加而膨胀

一个 Hook 系统的演进路径是可以预测的：从最初只支持 shell 命令，到有人要求"能不能用 AI 判断一下"，再到"能不能通知远程服务"。每次新增执行器类型，如果都要修改主调度函数，就会引入两个风险：原有执行器的测试需要全部重跑，以及调度函数本身的复杂度不断累积。

Claude Code 的解决方案是在 src/utils/hooks.ts:1952 的 executeHooks 中实现纯粹的类型分发：

// src/utils/hooks.ts:1952-1976（简化）
async function* executeHooks({
  hookInput,
  toolUseID,
  signal,
  timeoutMs = TOOL_HOOK_EXECUTION_TIMEOUT_MS,  // 默认 10 分钟
  toolUseContext,
  messages,
  ...
}): AsyncGenerator<AggregatedHookResult> {
  // 安全门禁：任何原因禁用时直接返回
  if (shouldDisableAllHooksIncludingManaged()) return
  if (isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)) return
  if (shouldSkipHookDueToTrust()) return

  // 查找匹配的 hooks 列表
  const matchingHooks = await getMatchingHooks(...)
  if (matchingHooks.length === 0) return

  // 并行运行所有匹配的 hooks，每个 yield 结果
  for each matchingHook:
    → if type === 'callback' → executeHookCallback
    → if type === 'prompt'   → execPromptHook
    → if type === 'agent'    → execAgentHook
    → if type === 'http'     → execHttpHook
    → default (type === 'command') → 内联执行 shell 命令
}

源码参考： src/utils/hooks.ts:1952

executeHooks 被声明为 async function*（异步生成器），每处理完一个 hook 就 yield 其结果，而不是等所有 hook 都完成再一次性返回。这让 UI 层能逐步接收和展示 hook 的进度，而不是在等待期间一片空白。

TOOL_HOOK_EXECUTION_TIMEOUT_MS = 10 * 60 * 1000（10 分钟）是 command 和 http 执行器的默认超时，而 prompt 和 agent 执行器有各自独立的默认值。这正是多态的体现：每种执行器可以根据自己的特性选择合适的超时策略。

源码实例 1：execAgentHook——最复杂的执行器

execAgentHook 是四种执行器中复杂度最高的：它不是执行一个简单的命令，而是启动一个完整的多轮 AI 对话，使用工具集去验证某个条件是否满足。

我们来看它的核心设计（src/utils/hooks/execAgentHook.ts:36）：

源码参考： src/utils/hooks/execAgentHook.ts:36

注意 transcriptPath 的选择逻辑（第 54 行）：如果当前运行在子 Agent 上下文（toolUseContext.agentId 非空），就使用子 Agent 的 transcript 路径；否则使用主会话的 transcript 路径。这实现了transcript 隔离——hook agent 的执行历史不会混入触发它的父会话记录，保持了每层记录的独立性。

addArgumentsToPrompt（第 60 行）将 $ARGUMENTS 替换为触发事件的 JSON 输入。这是 hook 配置中唯一的动态插值点：hook 作者可以在 prompt 模板里写 $ARGUMENTS，让 agent 在运行时知道"这次触发是因为什么事件、涉及哪个工具调用"。

注释中明确说明"直接创建用户消息，绕过 processUserInput"的原因——如果走正常的用户输入处理路径，会再次触发 UserPromptSubmit hook，导致 hook → 触发 hook → 再次触发 hook 的无限递归。这是递归防护的设计决策，与第 22 章的 AutoCompact 递归防护（querySource === 'compact' 早返回）是同样的模式思路。

工具白名单过滤（第 98 行）是 execAgentHook 特有的安全措施：

源码参考： src/utils/hooks/execAgentHook.ts:98

ALL_AGENT_DISALLOWED_TOOLS 定义了 hook agent 不能使用的工具集，核心目的是防止 hook agent 启动新的子 Agent 或进入 plan 模式。如果不做这个限制，一个 stop hook 的 agent 可以无限级联地生成子 Agent，让 hook 成为逃逸到主 Agent 执行框架外部的后门。

execAgentHook 的默认超时是 60 秒（hook.timeout ? hook.timeout * 1000 : 60000），比默认的 10 分钟 command 超时短得多。为什么 agent 执行器的超时更短？因为 agent hook 的设计意图是"快速验证一个条件"，而不是执行长时间的任务——60 秒对于一个目标明确的验证 agent 来说足够，也防止一个意外陷入循环的 hook agent 长期占用资源。

源码实例 2：execHttpHook——最轻量、安全边界最精细的执行器

execHttpHook 是四种执行器中逻辑最简单的——一个 JSON POST——但它的安全策略是四者中最精细的。这个对比本身就值得思考：越简单的执行器，反而越需要精细的安全边界。

URL 白名单策略由 getHttpHookPolicy（src/utils/hooks/execHttpHook.ts:49）提供：

// src/utils/hooks/execHttpHook.ts:49-57
function getHttpHookPolicy(): {
  allowedUrls: string[] | undefined
  allowedEnvVars: string[] | undefined
} {
  const settings = settingsModule.getInitialSettings()
  return {
    allowedUrls: settings.allowedHttpHookUrls,
    allowedEnvVars: settings.httpHookAllowedEnvVars,
  }
}

源码参考： src/utils/hooks/execHttpHook.ts:49

allowedUrls 有三种语义：

allowedUrls 值含义
undefined不限制 URL（未配置白名单）
[]（空数组）阻断所有 HTTP hook 请求
["https://api.example.com/*"]只允许匹配模式的 URL

这三态语义来自 execHttpHook 的判断逻辑：

// src/utils/hooks/execHttpHook.ts:135-145（简化）
const policy = getHttpHookPolicy()
if (policy.allowedUrls !== undefined) {
  const matched = policy.allowedUrls.some(p => urlMatchesPattern(hook.url, p))
  if (!matched) {
    return { ok: false, body: '', error: `HTTP hook blocked: ${hook.url} does not match any pattern in allowedHttpHookUrls` }
  }
}

源码参考： src/utils/hooks/execHttpHook.ts:135

只有当 allowedUrls !== undefined 时才做白名单检查——undefined 表示用户没有配置白名单，这时不限制；[] 表示用户明确配置了一个空白名单，这时阻断所有请求。这个三态设计让"未配置"和"明确禁止"有不同的语义，而不是把两者都当成"不允许"。

urlMatchesPattern（第 64 行）使用 * 作为通配符，与 MCP 服务器 allowlist 相同的模式语义——用户只需学习一套白名单写法。

环境变量插值（interpolateEnvVars，第 89 行）解决了一个实际问题：HTTP hook 通常需要携带认证 token（如 Authorization: Bearer $MY_TOKEN），但把 token 硬编码在 settings.json 中是安全反实践。插值让配置文件保存模板，运行时从环境变量读取实际值。

// src/utils/hooks/execHttpHook.ts:89-110（简化）
function interpolateEnvVars(
  value: string,
  allowedEnvVars: ReadonlySet<string>,
): string {
  const interpolated = value.replace(
    /\$\{([A-Z_][A-Z0-9_]*)\}|\$([A-Z_][A-Z0-9_]*)/g,
    (_, braced, unbraced) => {
      const varName = braced ?? unbraced
      if (!allowedEnvVars.has(varName)) {
        // 不在白名单的变量名被替换为空字符串，而非保留原文或报错
        return ''
      }
      return process.env[varName] ?? ''
    },
  )
  return sanitizeHeaderValue(interpolated)  // 移除 CR/LF/NUL 防止 HTTP 头注入
}

源码参考： src/utils/hooks/execHttpHook.ts:89

注意"不在白名单的变量名被替换为空字符串"的决策——而不是保留 $VAR_NAME 原文。如果保留原文，恶意的 hook 配置可以通过 $UNKNOWN_VAR 绕过检查，在请求头中携带原始模板字符串来探测系统支持哪些变量。替换为空字符串确保白名单外的变量名完全消失，不留任何可利用的信息。

最后，sanitizeHeaderValue 移除 CR（\r）、LF（\n）和 NUL 字节——这防止了 CRLF 注入攻击：如果环境变量值包含 token\r\nX-Evil: 1，不做清洗就会在 HTTP 请求中注入一个恶意头。

模式剖析：多态执行器的四个关键组成部分

多态执行器模式有四个相互协作的设计约束：

1. 统一入参签名（Unified Parameter Interface）：四种 exec*Hook 函数共享一个基础参数集——hook（hook 配置对象）、hookEvent（事件名）、jsonInput（序列化的事件输入）、signal（取消信号）。额外参数（如 toolUseContext、messages）按需在 prompt 和 agent 执行器上增加。统一签名让 executeHooks 的分发代码保持简洁，不需要为不同类型准备不同的参数包装。

2. 纯分发主函数（Pure Dispatch Entry）：executeHooks 只做三件事：安全检查（shouldSkipHookDueToTrust、shouldDisableAllHooks）、查找匹配 hook（getMatchingHooks）、按类型分发。它不包含任何执行器的具体逻辑。这实现了开放-封闭原则：新增执行器类型（如未来的 grpc 类型）只需新增 execGrpcHook.ts 并在 dispatch 中加一个分支，不改动已有执行器代码。

3. 执行器独立安全策略（Per-Executor Security）：每种执行器根据自己的风险面实现独立的安全策略——command 依赖工作区信任门禁；agent 过滤工具白名单 + 防递归；http 做 URL 白名单 + 环境变量插值 + HTTP 头注入防护。统一的安全层（shouldSkipHookDueToTrust）是所有执行器共享的第一道防线，但每种执行器还有自己的纵深防御。

4. 递归防护（Recursion Guard）：prompt 和 agent 执行器都绕过了 processUserInput，直接构造用户消息，避免 hook → UserPromptSubmit → hook 的无限递归。这是设计文档（注释）明确说明的刻意决策，不是偶然。

权衡与局限

权衡 1：prompt vs agent 执行器的选择边界

prompt 执行器（30 秒超时）和 agent 执行器（60 秒超时）都调用 AI，但语义不同：prompt 是单轮评估（queryModelWithoutStreaming，只有一次 AI 调用），agent 是多轮验证（最多 50 轮对话，MAX_AGENT_TURNS = 50）。用 agent 执行器做本来 prompt 执行器就能完成的简单判断，会浪费 token 并增加延迟；用 prompt 执行器做需要读文件、运行命令才能判断的复杂条件，又会得到不可靠的结果（AI 没有工具执行能力，只能靠臆测）。这个边界是隐性的——没有代码强制用户选择合适的执行器，依赖用户的理解。

权衡 2：http 执行器的 URL 白名单默认关闭

allowedUrls: undefined（未配置）意味着 http 执行器默认不限制 URL——任何 hook 配置都可以向任意 URL 发请求。这在方便性和安全性之间选择了方便性：强制要求用户配置白名单会让首次使用摩擦过高，但默认无限制在恶意 hook 场景下存在 SSRF 风险。这个 trade-off 依赖于"工作区信任门禁"作为第一道防线——只有用户明确信任了当前工作区，http hook 才会执行。

权衡 3：execAgentHook 的工具过滤粒度

ALL_AGENT_DISALLOWED_TOOLS 是一个全局黑名单，所有 agent hook 都使用相同的过滤规则。这无法针对不同 hook 配置允许不同的工具集（如"只有这个 hook 可以用 BashTool"）。更细粒度的工具权限需要在 hook 配置中增加 allowedTools 字段，但当前实现选择了全局黑名单的简单方案（推断：这是为了避免用户在 hook 配置中授予过于宽泛的工具权限）。

权衡 4：异步生成器的背压问题

executeHooks 每 yield 一个结果，调用方必须消费这个结果才能触发下一个执行器。如果调用方处理结果很慢，后续执行器会等待。这在理论上形成背压，但在实践中不是问题——目前每个 hook 的结果都被立即消费（UI 展示或聚合处理），没有延迟消费的场景（推断）。

与已知模式的对话

与 GoF 命令模式（Command Pattern）：命令模式将操作封装为对象，支持排队、撤销和重放。execHook 函数也是"将执行行为封装为独立函数"，但差异在于：命令模式通常支持 undo()（撤销），而 hook 执行器是单向的，没有撤销语义——一个 PostToolUse agent hook 执行后无法撤销其 AI 调用。另外，命令模式的"命令对象"通常是动态创建的，而 execHook 是静态分发到预定义函数，更接近函数分派而非命令对象。

与 GoF 策略模式（Strategy Pattern）：策略模式是与本模式最接近的 GoF 模式。策略模式定义算法族，每种算法封装在独立类/函数中，可以运行时替换。exec*Hook 函数正是"执行策略族"的实现。差异在于：策略模式通常同时只使用一个策略（Context 持有一个策略实例），而 executeHooks 支持同时触发多个执行器（一个 PreToolUse 事件可以同时触发 command hook 和 http hook）。这是本模式超越标准策略模式的关键扩展。

与中间件链（Middleware Chain，如 Koa）：Koa 的中间件链通过 next() 形成洋葱模型，前一个中间件可以决定是否调用后一个，中间件之间可以共享 ctx。exec*Hook 执行器没有"next"概念——每个执行器独立运行，不感知其他执行器的存在。这简化了模型，但也意味着执行器之间无法传递状态（如"前一个 hook 的结果影响下一个 hook 的参数"）。

模式提炼

多态执行器（Polymorphic Executor）

解决的问题：扩展系统需要支持多种执行方式，但主调度逻辑不应随执行器类型增加而膨胀，且不同执行器需要独立的安全策略。

核心做法：每种执行类型有独立的 exec*Hook 函数，主调度（executeHooks）只做安全检查 + 类型分发，不包含执行器逻辑；执行器间通过统一的参数签名（hook, hookEvent, jsonInput, signal）松耦合。

前置条件：执行器有可以统一的基础参数接口；主调度不需要了解执行器的内部实现细节。

源码证据：src/utils/hooks.ts:2224（if prompt → execPromptHook，if agent → execAgentHook，if http → execHttpHook）；src/utils/hooks/execAgentHook.ts:36（agent 执行器的独立函数和工具白名单）

防递归直接消息创建（Anti-Recursion Direct Message）

解决的问题：Hook 执行器调用 AI 时，如果走正常的用户输入路径，会再次触发 UserPromptSubmit hook，导致 hook → hook 的无限递归。

核心做法：在 prompt 和 agent 执行器中，直接通过 createUserMessage 构造消息（execAgentHook.ts:60、execPromptHook.ts:35），绕过 processUserInput，从源头切断递归链路。

前置条件：系统有可区分"用户发起的输入"和"执行器内部构造的消息"的机制。

源码证据：src/utils/hooks/execAgentHook.ts:60（注释说明"Create user message directly — no need for processUserInput which would trigger UserPromptSubmit hooks and cause infinite recursion"）

你能做什么

  • 将每种执行类型封装为独立的 exec* 函数，主调度只做类型分发。新增执行类型时只需新增函数和一个 if 分支，不改动已有执行器逻辑，符合开放-封闭原则。

  • 为 HTTP 执行器实现三态 URL 白名单（undefined=不限、[]=全部拒绝、[patterns]=匹配允许），区分"用户未配置"和"用户明确禁止"的语义，而不是把两者都视为拒绝。

  • 为 AI 执行器（prompt/agent）设计防递归机制：在 hook 内部调用 AI 时，绕过会触发同类 hook 的输入处理路径，直接构造消息传给模型。注释中明确说明为什么绕过，防止未来维护者"修复"这个"绕过"。

  • 为 agent 执行器维护工具黑名单（ALL_AGENT_DISALLOWED_TOOLS），防止 hook agent 启动子 Agent 或进入特殊模式，限制其权限到"验证条件"的最小必要集合。

  • 在 hook prompt 模板中支持 $ARGUMENTS 插值（addArgumentsToPrompt），让 hook 配置者可以在 prompt 里引用触发事件的具体内容，而不是每个 hook 都硬编码"不知道触发了什么"的通用提示词。

  • 对头部值中的环境变量插值实施双重防护：先做白名单过滤（只替换显式允许的变量），再做 CRLF 净化（移除 \r\n\0）。白名单防止变量名探测，净化防止头注入——两道防线分别应对不同的攻击向量。

  • 用异步生成器（async function*）实现多执行器的顺序调度，让调用方能逐步接收每个执行器的结果，而不是等所有执行器完成后才能处理。这在 UI 层特别有价值——可以实时显示"hook 正在执行"的进度。

四种执行器共享了一套钩子触发框架，而这套框架中异步注册和并发控制的具体实现，由 AsyncHookRegistry 负责（详见第 26 章）。钩子配置如何在进程启动时被加载、隔离并注入到执行上下文中，则是第 27 章 hooksConfigSnapshot 的主题（详见第 27 章）。
