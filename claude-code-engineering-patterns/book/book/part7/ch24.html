Claude Code 工程模式

第 24 章：27 个事件的生命周期地图——Hooks 全景

"可观测性不是事后加上去的，而是在系统的每个决策点预留的窗口。"

每当 AI 执行一次工具调用，Claude Code 至少触发 3 个精心命名的事件：调用前的 PreToolUse、调用成功后的 PostToolUse、调用失败后的 PostToolUseFailure。将视野扩展到整个会话——从会话启动到子 Agent 管理，从权限决策到配置变更——系统一共预埋了 27 个事件节点，构成了一张完整的 Agent 生命周期地图。

这 27 个事件不是随意堆砌的通知信号，而是按照"可独立拦截"的原则精心划分的：每个节点对应一种特定的执行时机，每种时机允许外部代码注入决策或收集数据。这就是可观测 Agent 事件总线（Observable Agent Event Bus）模式——用穷举关键时刻的方式，把一个原本不透明的 AI 行为系统变成可审计、可拦截的编程接口。

理解这 27 个事件的分组逻辑，你就能在自己的 Agent 系统中设计一套"每个关键时刻都可以拦截"的钩子体系，而不是依赖日志或 monkey-patching 去感知 Agent 行为。

问题：AI Agent 的行为是不透明的

当用户让 AI 帮忙"整理一下代码目录"时，AI 可能会删除它认为不需要的文件。你怎么阻止它？

一个没有钩子系统的 Agent 框架给出的答案只有两种：要么修改框架源码插入检查点，要么在工具执行后检查结果并亡羊补牢。两种方案都代价高昂——前者破坏可维护性，后者为时已晚。

钩子（Hook）事件的本质是一种"可中断的观察点"：在系统执行路径的关键节点，预先定义命名信号，允许外部代码注册处理器。与普通日志的区别在于，部分钩子事件支持阻断（block）——处理器可以返回拒绝决策，阻止后续操作继续。

源码实例 1：工具调用生命周期——PreToolUse 的"拦截-执行-汇报"切面

工具调用是 AI 行为中风险最高的环节。我们来看这三个事件如何形成一个完整的切面。

createBaseHookInput 是所有钩子事件的通用输入基础，定义在 src/utils/hooks.ts:301：

// src/utils/hooks.ts:301-330（简化）
export function createBaseHookInput(
  permissionMode?: string,
  sessionId?: string,
  agentInfo?: { agentId?: string; agentType?: string },
): {
  session_id: string
  transcript_path: string
  cwd: string
  permission_mode?: string
  agent_id?: string
  agent_type?: string
} {
  const resolvedSessionId = sessionId ?? getSessionId()
  // agent_type: 子 Agent 的类型优先于 --agent flag，
  // hooks 用 agent_id 存在与否区分子 Agent 和主线程调用
  const resolvedAgentType = agentInfo?.agentType ?? getMainThreadAgentType()
  return {
    session_id: resolvedSessionId,
    transcript_path: getTranscriptPathForSession(resolvedSessionId),
    cwd: getCwd(),
    permission_mode: permissionMode,
    agent_id: agentInfo?.agentId,
    agent_type: resolvedAgentType,
  }
}

源码参考： src/utils/hooks.ts:301

这个函数揭示了所有钩子事件共享一个最小公共接口的设计原则：session_id（可追踪性）、transcript_path（关联到对话记录）、cwd（当前工作目录），加上可选的 permission_mode 和 agent_id。无论是 PreToolUse 还是SessionEnd，基础字段都相同——钩子处理器无需关心当前是哪个具体事件就能实现通用的日志记录或追踪逻辑。

agent_id 字段值得特别注意：注释中说"hooks 用 agent_id 存在与否区分子 Agent 和主线程调用"。这意味着钩子处理器可以通过检查 agent_id 是否存在，判断当前的工具调用是由主 Agent 还是子 Agent 发起的，从而实现不同的处理策略。

getMatchingHooks 负责按事件名查找已注册的钩子处理器（src/utils/hooks.ts:1603）。对于工具相关事件，它使用 tool_name 作为匹配查询：

// src/utils/hooks.ts:1617-1625
switch (hookInput.hook_event_name) {
  case 'PreToolUse':
  case 'PostToolUse':
  case 'PostToolUseFailure':
  case 'PermissionRequest':
  case 'PermissionDenied':
    matchQuery = hookInput.tool_name  // 按工具名匹配
    break
  ...
}

源码参考： src/utils/hooks.ts:1617

这一设计让钩子注册者可以精确控制"我只对 BashTool 的调用感兴趣"，而不必接收所有工具的事件然后自己过滤。不同事件使用不同的 matchQuery：SubagentStart/Stop 用 agent_type，SessionEnd 用 reason，FileChanged 用 basename(file_path)——每种事件的"可区分维度"不同，匹配字段也就不同。

PreToolUse 的钩子响应格式（来自 src/types/hooks.ts 的 syncHookResponseSchema）支持多种操作：

  • decision: 'approve' | 'block'：批准或阻断工具执行
  • hookSpecificOutput.updatedInput：修改传给工具的输入参数
  • hookSpecificOutput.permissionDecision：影响后续的权限决策
  • hookSpecificOutput.additionalContext：向 AI 注入额外的上下文信息

源码实例 2：会话与子 Agent 生命周期——超时差异化设计

会话生命周期事件与工具调用事件有一个根本区别：SessionEnd 必须等待钩子完成，否则会话关闭可能导致数据丢失。这一差异体现在独立的超时配置上。

getSessionEndHookTimeoutMs 是 SessionEnd 专属的超时获取函数（src/utils/hooks.ts:176）：

// src/utils/hooks.ts:175-182
const SESSION_END_HOOK_TIMEOUT_MS_DEFAULT = 1500
// ...
export function getSessionEndHookTimeoutMs(): number {
  const raw = process.env.CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS
  const parsed = raw ? parseInt(raw, 10) : NaN
  return Number.isFinite(parsed) && parsed > 0
    ? parsed
    : SESSION_END_HOOK_TIMEOUT_MS_DEFAULT
}

源码参考： src/utils/hooks.ts:175

默认超时是 1,500 毫秒，并且支持通过环境变量 CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS 覆盖。为什么 SessionEnd 需要独立的超时值，而不是复用其他事件的超时配置？

在会话正常关闭的场景下，系统正在清理状态、关闭文件句柄、写入最终记录。如果 SessionEnd 的钩子（比如"会话结束后同步记忆到远程服务器"）超时了，有两种处理策略：等待更长时间（让数据同步完成，代价是会话关闭被延迟）或直接跳过（代价是数据丢失）。1,500ms 的默认值是一个在"不让用户等太久"和"给钩子足够时间完成写操作"之间的折中——比大多数工具调用的 hook 超时更短，但足以完成一次网络请求。

钩子系统还内置了一个安全门禁：shouldSkipHookDueToTrust（src/utils/hooks.ts:286）：

// src/utils/hooks.ts:286-298
export function shouldSkipHookDueToTrust(): boolean {
  // 非交互模式（SDK）中，信任是隐式的——始终执行
  const isInteractive = !getIsNonInteractiveSession()
  if (!isInteractive) {
    return false
  }

  // 交互模式中，所有 hooks 都需要信任
  const hasTrust = checkHasTrustDialogAccepted()
  return !hasTrust
}

源码参考： src/utils/hooks.ts:286

这个函数区分了两种运行模式：

  • 非交互（SDK）模式：调用者是已知的程序，信任是隐式的，不需要用户确认
  • 交互（CLI）模式：用户可能运行了来源不明的 .claude/settings.json 中的钩子配置，必须通过信任对话框后才能执行

为什么需要这个门禁？因为钩子是可以执行任意 shell 命令的——如果一个被分享的项目目录中包含了恶意的 hook 配置，未经确认直接执行会带来安全风险。shouldSkipHookDueToTrust 是钩子系统的安全边界：执行用户明确信任的钩子，跳过不明来源的钩子。

子 Agent 事件（SubagentStart/SubagentStop）则让钩子系统具备了层次结构的可观测性。从 getMatchingHooks 的匹配逻辑可以看到，这两个事件以 agent_type 作为匹配条件——注册者可以选择"只监控 task 类型的子 Agent 启动"，而不必接收所有子 Agent 的事件。

模式剖析：可观测 Agent 事件总线的核心结构

可观测 Agent 事件总线模式由四个相互协作的组成部分构成：

1. 穷举式事件枚举（Exhaustive Event Enumeration）：HOOK_EVENTS 以字符串字面量联合类型定义，是一个封闭集合。外部代码不能"动态注册新的事件类型"——只能注册对已有 27 个事件中某个的处理器。这个设计选择牺牲了扩展性（添加新事件需要修改枚举），换取了编译期完整性：任何对事件名的拼写错误都会在编译期被发现。

2. 共同基础接口（Common Base Interface）：createBaseHookInput 保证所有事件都携带 session_id、transcript_path、cwd——这三个字段让每个钩子调用都可以被追溯到具体的会话和工作目录，是可观测性的基础设施。

3. 精细匹配查询（Fine-grained Match Query）：getMatchingHooks 的 switch-case 逻辑为每种事件选择最合适的匹配维度——工具事件用 tool_name，文件事件用文件名，子 Agent 事件用 agent_type。这让钩子注册者能够精确表达"我关心什么"的意图。

4. 阻断能力分层（Blocking Capability Stratification）：只有"执行前"事件（PreToolUse、PermissionRequest、Elicitation）支持阻断响应（decision: 'block'）；"执行后"事件只能附加上下文。这个设计反映了一个工程现实：阻断必须在不可逆操作发生前进行，事后只能观测和修复。

权衡与局限

权衡 1：27 个事件的维护成本

HOOK_EVENTS 是一个硬编码的字符串字面量联合类型。每次系统新增功能（如未来的 MemoryWrite 或 SubagentMessage 事件），都需要修改这个数组，同时更新所有相关的 switch-case（如 getMatchingHooks）。当前 27 个事件的设计覆盖了几乎所有可观测节点，但随着功能演进，枚举膨胀是不可避免的（推断）。与"开放式事件注册"（允许插件定义新事件类型）相比，封闭枚举的代价是更高的维护成本，收益是更强的类型安全。

权衡 2：同步拦截与性能

PreToolUse 等支持 block 的事件是同步拦截的——工具执行必须等待钩子处理器返回决策。如果钩子处理器调用了远程服务（如将操作发送到审计后端等待批准），这会直接增加工具调用的延迟。异步钩子（async: true 响应格式）可以绕过这一限制，但异步钩子无法返回 block 决策。这是一个根本性的 trade-off：实时拦截需要同步等待，低延迟需要异步执行，两者不可兼得。

权衡 3：SessionEnd 的 1,500ms 默认超时

1,500ms 对于大多数网络请求是足够的，但如果钩子需要执行耗时操作（如同步大量记忆文件），用户就需要手动配置 CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS。这个配置项的存在本身就说明：设计者知道这个默认值不能满足所有场景，预留了逃生通道。

权衡 4：shouldSkipHookDueToTrust 的粒度

当前信任检查是全局的——要么信任所有钩子，要么跳过所有钩子。无法表达"信任来自官方插件的钩子，但不信任项目目录里的自定义钩子"（推断）。更细粒度的信任策略（如基于来源的白名单）会增加安全模型的复杂性，当前实现选择了简单但偏保守的全局开关。

与已知模式的对话

与 GoF 观察者模式（Observer Pattern）：两者都是发布-订阅机制。根本差异在于介入能力：标准观察者（GoF）只能被动接收状态变化通知，无法阻止或修改被观察对象的行为；Claude Code 的 PreToolUse 钩子可以返回 decision: 'block' 中止工具执行，PostToolUse 可以返回 updatedInput 修改工具输出传给 AI 的内容。这使本模式更接近**拦截器（Interceptor）**而非观察者。

与 EIP 事件驱动消费者（Event-Driven Consumer）：事件驱动消费者是异步解耦模式——生产者发布事件，消费者异步处理，两者通过消息队列解耦，互不等待。Claude Code 的同步 hook 与之相反：在决策性事件（PreToolUse）上，系统必须等待钩子返回才能继续执行。异步钩子（async: true）更接近 EIP 消费者模式，但以失去阻断能力为代价。

与面向切面编程 AOP（Aspect-Oriented Programming）：本模式与 AOP 最为接近。AOP 的核心概念是"切入点"（Pointcut，定义在哪里切入）+"通知"（Advice，切入后执行什么）。HOOK_EVENTS 枚举是系统预定义的所有切入点，钩子处理器是对应的通知。差异在于：AOP 通常在编译期或类加载期织入，运行时没有性能开销；Claude Code 的钩子是运行时动态查找（getMatchingHooks），有额外的查找和执行开销。

模式提炼

可观测 Agent 事件总线（Observable Agent Event Bus）

解决的问题：AI Agent 的行为不可审计、不可拦截，外部系统无法在不修改框架源码的情况下介入关键操作。

核心做法：在执行路径的关键节点预埋 27 个命名事件，使用字符串字面量联合类型保证编译期完整性，允许外部注册处理器；其中"执行前"事件支持阻断，"执行后"事件支持附加上下文。

前置条件：Agent 框架有明确的执行路径；事件节点的数量有限且稳定（不适合需要频繁新增事件类型的高速演进场景）。

源码证据：src/entrypoints/sdk/coreTypes.ts:25（HOOK_EVENTS 27 个事件，as const 字面量联合）；src/utils/hooks.ts:1603（getMatchingHooks，按事件精细匹配）

事件超时差异化（Event Timeout Differentiation）

解决的问题：不同生命周期节点对"等待钩子完成"的容忍度不同——会话结束需要给写操作足够时间，而普通工具调用需要快速响应。

核心做法：为关键事件（SessionEnd）配置独立的超时常量（SESSION_END_HOOK_TIMEOUT_MS_DEFAULT = 1500）和环境变量覆盖机制（CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS），而非全局统一超时。

前置条件：不同事件有明确不同的时序语义（"必须等待完成"vs"可以超时跳过"）。

源码证据：src/utils/hooks.ts:175（SESSION_END_HOOK_TIMEOUT_MS_DEFAULT = 1500）；src/utils/hooks.ts:176（getSessionEndHookTimeoutMs，支持环境变量覆盖）

信任门禁（Trust Gating）

解决的问题：钩子可以执行任意 shell 命令，项目目录中的恶意钩子配置在未经确认的情况下会造成安全风险。

核心做法：在执行任何钩子前检查信任状态（shouldSkipHookDueToTrust），非交互模式（SDK）隐式信任，交互模式必须通过显式信任对话框；信任未建立时跳过所有钩子。

前置条件：有可区分"交互"和"非交互"的运行模式；有持久化的信任状态（checkHasTrustDialogAccepted）。

源码证据：src/utils/hooks.ts:286（shouldSkipHookDueToTrust，根据会话类型和信任状态决定是否跳过）

你能做什么

  • 用字符串字面量联合类型定义 Agent 事件枚举（as const），而非数字 ID 或普通字符串。编译期完整性检查能在事件名拼写错误时立即报错，而不是运行时静默丢失钩子。

  • 为工具调用设计"前-后-失败"三元事件对（PreToolUse/PostToolUse/PostToolUseFailure）。成功和失败是独立事件而非参数变体，让钩子注册者可以精确表达"只关心失败的 BashTool 调用"，而不必在通用处理器中自己判断。

  • 为不同事件类型选择最有意义的匹配维度。工具事件用工具名匹配，文件事件用文件名匹配，子 Agent 事件用 agent_type 匹配——让注册者能精确过滤，而不是接收所有事件后自己过滤。

  • 为关键生命周期事件（如会话结束）配置独立的超时值，区分"数据写入必须完成"和"可以超时跳过"的钩子。一个全局超时无法同时满足高响应性和数据完整性的要求。

  • 为所有钩子事件定义共同基础接口（session_id、工作目录、时间戳），保证每个钩子调用都可以被追溯到具体的执行上下文。钩子处理器越通用，基础字段就越重要。

  • 在钩子执行前实现信任门禁，区分受信任的内置钩子和来源不明的外部钩子。特别是在支持插件的系统中，不经验证直接执行所有钩子是一个常见的安全漏洞。

  • 将"阻断能力"限制在执行前事件。不要试图在事件总线上实现"撤销"（执行后 block）——这会引入复杂的状态回滚逻辑。清晰区分"可阻断的执行前事件"和"只可观测的执行后事件"，让系统行为更可预测。

27 个事件定义了"能在哪里介入"，而介入后的实际执行——钩子命令如何触发、command/prompt/agent 三种执行器如何工作——是第 25 章的主题（详见第 25 章）。钩子的异步注册和并发控制机制，则由 AsyncHookRegistry 管理（详见第 26 章）。
