# 序言：一个源码侦探的自白

Claude Code 的 npm 包里意外留了一个 60MB 的 source map 文件。1,902 个源文件、51 万行 TypeScript 代码，全部暴露。

第一次打开 `src/main.tsx`，映入眼帘的不是入口函数，而是一段工程备忘录——注释里写着"利用 import 求值期间约 135ms 的等待窗口并行执行 I/O 操作"。我本来只想搞清楚"这个 CLI 到底怎么跑起来的"，但顺着 `launchRepl()` 往下追了三个文件，意识到自己掉进了一个远比文档描述更精密的世界。

**这不是一个 CLI。这是一套 Agent Harness。**

「Harness」是工程术语——把不可控的外部系统包裹在可控接口内的框架结构。Claude Code 的 Harness 层有十个独立子系统、58 个工具目录、20+ 个 Hook 事件、7 种任务类型、三种多智能体后端——每一个数字背后，都是一个小系统在功能压力下被迫膨胀的痕迹。Hook 从 3 个事件增长到 20+ 个，Task 从 1 种扩展到 7 种，Swarm 从单进程演化出三种后端——**没有一次是推倒重来**。教科书教的是"应该怎么设计"，源码展示的是"实际上怎么演化"。本书就是沿着这些演化伤疤做的一次完整追踪。

---

## 这本书适合谁

**主要读者**：正在构建 AI Agent 系统的工程师和架构师。如果你在问"工具调用怎么设计才能可扩展？权限系统怎么避免成为瓶颈？多个 Agent 如何安全协作？"——本书提供了一个 51 万行生产代码的真实样本。

**次要读者**：对 AI 工程底层机制感兴趣的开发者。Claude Code 是目前罕见的源码完整暴露的生产级 Agent Harness。

**不适合**：只想了解 Claude Code 使用方法的用户。本书没有操作指南，没有配置教程，只有工程决策的追踪和模式提炼。

---

## 如何阅读本书

全书 45 章正文，分 12 篇，遵循两条并行线索：

- **源码追踪线**：每章跟随一个具体的工程问题——"为什么 `QueryEngine` 在进入循环前先写转录文件？""为什么权限系统用三层而非一层？"——顺着源码追到设计决策的原因。
- **模式提炼线**：每章末尾提炼一个可命名的工程模式，配有源码锚点。第 44 章是全部 12 个模式的汇总图谱，可作为复习地图。

## 约定说明

**源码引用格式**：`src/相对路径:行号`，例如 `src/QueryEngine.ts:675`。引用前已验证行号对应 Claude Code v2.1.88（重建版）。

**术语标注**：技术术语首次出现时标注英文原文，例如「死代码消除（Dead Code Elimination, DCE）」；后续章节直接使用简称「DCE」。完整术语表见**附录 F**。

**推断标注**：无直接源码依据的设计意图推断，会明确标注「（推断）」，区别于有源码证据的断言。

**章节交叉引用**：已在其他章节详细分析过的内容，用「详见第 X 章」代替重复讲解。

**代码注释翻译**：英文注释翻译为中文，附原文（原文："..."）。

## 关于这个项目

本书的核心主张是：**Claude Code 是一个活体样本，展示了「从 CLI 工具到 Agent 平台」的架构跃迁如何在生产级代码中真实发生**。它不是从零设计的 Agent 框架，而是一个 CLI 工具在功能压力下被迫演化——每一个子系统都留下了演化的痕迹。

本书基于 Claude Code v2.1.88 的重建源码分析写成。书中所有设计决策均以源码为证据，不依赖官方文档或内部信息。提炼的 12 个工程模式是从源码中归纳的——它们是追踪之旅的工程结晶，不是 Anthropic 官方术语。
