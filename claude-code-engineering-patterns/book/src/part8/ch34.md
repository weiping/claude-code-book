# 第 34 章：PermissionRule 规则引擎——`allowedTools` 的 glob 匹配与优先级链

> "用户写下一行字符串，系统需要精确解析它——括号里可能有括号，星号不一定是通配符。"

---

用户在权限配置中写下 `Bash(npm install)` 就能允许 npm 安装命令，写下 `Bash(git:*)` 就能允许所有 git 命令。这看起来简单，但内部有两层非平凡的工程问题：**如果用户要允许包含括号的命令**（如 `python -c "print(1)"`），括号会破坏 `Bash(content)` 格式的解析；**`git:*` 和 `git *` 是不同的语法**，一个是旧版前缀匹配，一个是新版 glob 通配符，两者共存需要兼容层。

Claude Code 的规则解析引擎用两个文件解决了这两个问题：`permissionRuleParser.ts`（198 行）处理转义和解析，`shellRuleMatching.ts`（228 行）处理三种匹配语义（精确/前缀/通配符）。这是**声明式规则解析引擎**（Declarative Rule Parser）模式：格式即契约，规则字符串本身是完整的权限声明，不需要代码配置，解析层负责将用户输入的字符串精确还原为结构化决策。

---

## 问题：括号格式与转义的设计空间

`Bash(npm install)` 格式用括号分隔工具名和内容。简洁直观，但有一个问题：如果内容本身包含括号呢？

比如用户想允许 `python -c "print(1)"` 命令，规则字符串应该是 `Bash(python -c "print(1)")`——但这个字符串有三对括号，解析时如何知道哪个 `)` 是内容的结尾？

简单的 `indexOf(')')` 会在第一个 `)` 处截断，得到错误的解析结果。需要一个支持转义的解析机制。

`permissionRuleValueFromString` 的注释给出了三个示例（`src/utils/permissions/permissionRuleParser.ts:90`），清楚地展示了这个转义需求：

```
permissionRuleValueFromString('Bash')
// => { toolName: 'Bash' }

permissionRuleValueFromString('Bash(npm install)')
// => { toolName: 'Bash', ruleContent: 'npm install' }

permissionRuleValueFromString('Bash(python -c "print\\(1\\)")')
// => { toolName: 'Bash', ruleContent: 'python -c "print(1)"' }
```

第三个示例揭示了转义机制：括号在存储时用 `\(` 和 `\)` 转义，反斜杠本身用 `\\` 转义。解析时用 `findFirstUnescapedChar` 找到真正的格式括号，不被内容中的括号迷惑。

**图 34-1：规则字符串格式与匹配语义**

```
规则字符串格式：
  ┌──────────────────────────────────────────┐
  │  工具名 ( 内容（可含转义括号） )           │
  │  Bash  ( python -c "print\(1\)"  )       │
  │        ↑                          ↑      │
  │   第一个未转义的 (         最后一个未转义的 )│
  └──────────────────────────────────────────┘

匹配语义（shellRuleMatching.ts）：
  精确匹配：  Bash(npm install)    → 只匹配 "npm install"
  前缀匹配：  Bash(npm:*)          → 匹配 "npm" 开头的所有命令（legacy）
  通配符匹配：Bash(npm * install)  → 使用 glob，* 匹配任意序列
```

## 源码实例 1：permissionRuleParser——转义的对称性

`escapeRuleContent` 和 `unescapeRuleContent` 是规则字符串序列化/反序列化的核心（`src/utils/permissions/permissionRuleParser.ts:55`）：

```typescript
// src/utils/permissions/permissionRuleParser.ts:55-80（含注释）
/**
 * 转义顺序很重要：
 * 1. 先转义已有的反斜杠（\ -> \\）
 * 2. 再转义括号（( -> \(, ) -> \)）
 */
export function escapeRuleContent(content: string): string {
  return content
    .replace(/\\/g, '\\\\')  // 先转义反斜杠
    .replace(/\(/g, '\\(')   // 再转义开括号
    .replace(/\)/g, '\\)')   // 再转义闭括号
}

/**
 * 反转义顺序（与转义相反）：
 * 1. 先反转义括号（\( -> (, \) -> )）
 * 2. 再反转义反斜杠（\\ -> \）
 */
export function unescapeRuleContent(content: string): string {
  return content
    .replace(/\\\(/g, '(')   // 先反转义括号
    .replace(/\\\)/g, ')')
    .replace(/\\\\/g, '\\')  // 再反转义反斜杠
}
```

**源码参考：** `src/utils/permissions/permissionRuleParser.ts:55`

注释中明确强调"转义顺序很重要"——这是一个经典的双重转义陷阱。

**为什么反斜杠必须第一个被转义？** 设想内容是 `test\(value`（含一个反斜杠和一个括号）。如果先转义括号得到 `test\\(value`（反斜杠+括号 → 两个字符），再转义反斜杠会把这个新产生的 `\\` 也转义成 `\\\\`，最终变成 `test\\\\(value`——错误地双重转义了原来的反斜杠。正确顺序是先把所有已有的 `\` 变成 `\\`（扩充为转义形式），再把 `(` 变成 `\(`（用已安全的反斜杠前缀标记括号）。

反转义是完全对称的逆操作——先还原括号（因为 `\(` 中的反斜杠还是转义状态），再还原反斜杠（此时 `\\` 才能安全地变回 `\`）。顺序反转会导致 `\(` 中的 `\` 被先还原成没有意义的字符，破坏后续的括号还原。

`permissionRuleValueFromString` 的解析逻辑（`src/utils/permissions/permissionRuleParser.ts:93`）展示了如何用"找第一个未转义的 `(`"来处理嵌套括号：

```typescript
// src/utils/permissions/permissionRuleParser.ts:93-130（简化）
export function permissionRuleValueFromString(ruleString: string): PermissionRuleValue {
  // 找第一个未转义的开括号
  const openParenIndex = findFirstUnescapedChar(ruleString, '(')
  if (openParenIndex === -1) {
    return { toolName: normalizeLegacyToolName(ruleString) }  // 纯工具名
  }

  // 找最后一个未转义的闭括号
  const closeParenIndex = findLastUnescapedChar(ruleString, ')')
  if (closeParenIndex !== ruleString.length - 1) {
    return { toolName: normalizeLegacyToolName(ruleString) }  // 格式错误
  }

  const toolName = ruleString.substring(0, openParenIndex)
  const rawContent = ruleString.substring(openParenIndex + 1, closeParenIndex)

  // 特殊情况：空内容或单独的 * 视为纯工具名规则
  if (rawContent === '' || rawContent === '*') {
    return { toolName: normalizeLegacyToolName(toolName) }
  }

  return { toolName: normalizeLegacyToolName(toolName), ruleContent: unescapeRuleContent(rawContent) }
}
```

**源码参考：** `src/utils/permissions/permissionRuleParser.ts:93`

`findFirstUnescapedChar` 的逻辑是：找到目标字符后，向前数连续反斜杠的数量——如果是**偶数**（包括 0），字符未被转义（偶数个反斜杠两两抵消，最后没有剩余的转义符）；如果是**奇数**，字符被转义（奇数个反斜杠，最后一个充当转义符）。这个"奇偶判断法"是处理嵌套转义的标准技巧。

`normalizeLegacyToolName` 处理工具重命名的向后兼容：旧版 Claude Code 用 `Task` 作为工具名，现在叫 `Agent`。用户在旧版配置中写的规则 `Task(npm install)` 在新版中应该继续工作——`normalizeLegacyToolName('Task')` 返回 `'Agent'`，让旧规则透明地映射到新工具名。这个兼容层集中在解析函数中，不污染业务逻辑。

## 源码实例 2：shellRuleMatching——三种匹配语义与空字节哨兵

`shellRuleMatching.ts` 把内容字符串解析为三种匹配规则（`src/utils/permissions/shellRuleMatching.ts:25`）：

```typescript
// src/utils/permissions/shellRuleMatching.ts:25-36
export type ShellPermissionRule =
  | { type: 'exact'; command: string }    // 精确匹配
  | { type: 'prefix'; prefix: string }    // 前缀匹配（legacy :* 语法）
  | { type: 'wildcard'; pattern: string } // 通配符匹配（glob * 语法）
```

**源码参考：** `src/utils/permissions/shellRuleMatching.ts:25`

`parsePermissionRule` 按优先级依次判断是哪种类型（legacy 前缀 → glob 通配符 → 精确匹配）。`permissionRuleExtractPrefix`（第 43 行）处理 `:*` 语法：

```typescript
// src/utils/permissions/shellRuleMatching.ts:43-48
export function permissionRuleExtractPrefix(permissionRule: string): string | null {
  const match = permissionRule.match(/^(.+):\*$/)
  return match?.[1] ?? null  // "npm:*" → "npm"，否则 null
}
```

**源码参考：** `src/utils/permissions/shellRuleMatching.ts:43`

`:*` 是历史遗留语法——最初的规则系统用 `npm:*` 表示"允许所有 npm 子命令"。新版引入了 glob `*`，`npm *` 可以表达相同含义，且更灵活（`npm * install` 可以表达 npm 的任意子命令加上 install 参数）。两种语法共存，`permissionRuleExtractPrefix` 先检查是否是 `:*` 结尾的 legacy 格式。

`hasWildcards`（第 54 行）判断是否使用 glob `*`，同时跳过 legacy `:*` 语法：

```typescript
// src/utils/permissions/shellRuleMatching.ts:54-72
export function hasWildcards(pattern: string): boolean {
  if (pattern.endsWith(':*')) {
    return false  // legacy 语法，不是 glob 通配符
  }
  // 检查任何位置的未转义 * （同样用奇偶判断法）
  for (let i = 0; i < pattern.length; i++) {
    if (pattern[i] === '*') {
      let backslashCount = 0
      let j = i - 1
      while (j >= 0 && pattern[j] === '\\') { backslashCount++; j-- }
      if (backslashCount % 2 === 0) return true  // 未转义的 *
    }
  }
  return false
}
```

**源码参考：** `src/utils/permissions/shellRuleMatching.ts:54`

最精妙的实现细节在文件顶部的**空字节哨兵**（`src/utils/permissions/shellRuleMatching.ts:12`）：

```typescript
// src/utils/permissions/shellRuleMatching.ts:12-20
// 模块级的空字节哨兵，用于通配符模式转义——
// 模块级定义让 RegExp 对象只编译一次，而非每次权限检查时都编译。
const ESCAPED_STAR_PLACEHOLDER = '\x00ESCAPED_STAR\x00'
const ESCAPED_BACKSLASH_PLACEHOLDER = '\x00ESCAPED_BACKSLASH\x00'
const ESCAPED_STAR_PLACEHOLDER_RE = new RegExp(ESCAPED_STAR_PLACEHOLDER, 'g')
const ESCAPED_BACKSLASH_PLACEHOLDER_RE = new RegExp(ESCAPED_BACKSLASH_PLACEHOLDER, 'g')
```

**源码参考：** `src/utils/permissions/shellRuleMatching.ts:12`

`matchWildcardPattern` 需要把通配符 `*` 转换为正则表达式 `.*`，但 `\*`（用户转义的字面量星号）不应该被转换。直接的实现方案是在替换时做条件判断，但正则替换的条件逻辑很复杂。空字节哨兵提供了一个更简洁的两步方案：

1. **第一步**：把 `\*` 替换为 `\x00ESCAPED_STAR\x00`（一个普通的用户输入不可能包含的字符序列）
2. **第二步**：现在所有剩余的 `*` 都是未转义的通配符，可以安全地替换为 `.*`
3. **第三步**：把 `\x00ESCAPED_STAR\x00` 还原为 `\*`（正则表达式中的字面量星号）

哨兵字符是 `\x00`（null 字节）——用户的命令字符串不可能包含 null 字节，所以这个占位符不会与真实内容冲突。模块级定义（而非函数内定义）是性能优化：`RegExp` 对象只编译一次，每次权限检查时复用。

`matchWildcardPattern` 还有一个细节：当 pattern 以 ` *`（空格加单个通配符）结尾时，把最后的 `.*` 改为 `( .*)?`（可选的空格加任意字符），这样 `git *` 既能匹配 `git add` 也能匹配裸 `git`（没有参数）。这个行为与 legacy 前缀语法一致——`git:*` 通常意味着"允许 git 以及所有 git 子命令"。

## 模式剖析：声明式规则解析引擎的三个层次

**声明式规则解析引擎**由三个递进的层次构成：

**1. 格式层（Format Contract）**：`Tool(content)` 格式是规则的表示语言，转义机制（`\(`、`\)`、`\\`）让内容可以包含任意字符而不破坏格式。格式足够简洁（用户可以手写），也足够精确（机器可以准确解析）。

**2. 解析层（Parsing）**：`permissionRuleValueFromString` 将字符串分解为 `{toolName, ruleContent}` 结构体，处理转义和向后兼容（legacy 工具名别名）。解析层是纯函数——同样的输入永远产生同样的输出，无副作用，易于测试。

**3. 匹配层（Matching）**：`parsePermissionRule` 将内容字符串分类为三种规则类型（exact/prefix/wildcard），每种类型有不同的匹配语义。分类在使用前完成，运行时只需根据类型做对应的比较（字符串相等/前缀检查/正则匹配），不需要每次都重新解析规则字符串。

## 权衡与局限

**权衡 1：双重转义的认知负担**

用户手写规则时，如果内容包含括号，需要了解转义语法：`python -c "print(1)"` 要写成 `Bash(python -c "print\(1\)")`。对于不了解转义规则的用户，括号需求是一个隐藏的复杂度。UI 层（如"始终允许"对话框）通过自动调用 `escapeRuleContent` 隐藏了这个细节，但手动编辑配置文件的用户需要了解转义规则。

**权衡 2：`:*` 和 glob `*` 语法并存**

两种通配符语法（`npm:*` legacy 和 `npm *` new）表达相似但略有差异的语义，长期并存增加了用户的认知成本——用户需要知道哪个是"旧的"哪个是"新的"，以及它们在边缘情况下的行为差异（如 `git:*` vs `git *` 在匹配裸 `git` 时的行为）。当前 `matchWildcardPattern` 已经尝试统一两者的行为（trailing ` *` 变可选），但这是向后兼容的权宜之计，不是根本解决方案。

**权衡 3：正则表达式的安全性**

`matchWildcardPattern` 把用户输入的模式转换为正则表达式。虽然 `escapeRegExp` 已经转义了大部分特殊字符，但这条从用户输入到正则表达式的路径需要特别小心——正则 DoS（ReDoS）攻击可以通过精心构造的模式字符串让匹配时间呈指数增长。当前实现对输入做了充分的转义处理，但任何维护者修改这段代码时都需要了解这个风险。

## 与已知模式的对话

**与 Unix glob**：Unix glob 是最直接的参照——`*` 匹配任意字符序列，`?` 匹配单个字符。本系统的 glob 只使用 `*`（不支持 `?`），并且加了 `Tool(content)` 的工具名前缀层。另一个区别是 Unix glob 通常不支持转义（因为 shell 本身处理转义），本系统需要在字符串层面处理转义，因为规则字符串存储在 JSON 配置文件中，没有 shell 的转义层。

**与 SQL LIKE 语法**：SQL `LIKE` 用 `%` 表示任意字符序列，`_` 表示单个字符。本系统的 `:*` legacy 语法与 SQL LIKE 类似（都是特殊字符表示通配），但新 glob 语法更接近标准 glob。两个系统都需要转义机制处理字面量通配符，SQL 用 `ESCAPE` 子句，本系统用 `\` 前缀。

**与 DSL（领域特定语言）**：`Bash(npm install)` 是一个微型 DSL——完整语法是 `ToolName`（仅工具名）或 `ToolName(content)`（工具名加内容），content 中支持 `:*` 和 glob `*` 通配符，以及 `\(`、`\)`、`\\` 转义序列。这个语法刚好够用（覆盖了权限配置的所有需求），没有引入不必要的复杂性——如果加了条件表达式，就会变成需要完整解析器的真正 DSL，维护成本急剧增加。

## 模式提炼

### 声明式规则解析引擎（Declarative Rule Parser）

**解决的问题**：权限规则需要由用户以字符串形式配置，但用户输入的字符串可能包含格式分隔符（括号），需要精确的解析和转义机制。

**核心做法**：定义 `Tool(content)` 格式；`escapeRuleContent` 先转义反斜杠再转义括号；`unescapeRuleContent` 逆序反转；`findFirstUnescapedChar` 用奇偶计数法处理嵌套转义；`normalizeLegacyToolName` 在解析层集中处理工具名别名。

**前置条件**：规则格式简单且稳定（不需要完整解析器）；转义字符（`\`）不在合法内容中大量出现；有 legacy 兼容需求可以在解析函数入口处集中处理。

**源码证据**：`src/utils/permissions/permissionRuleParser.ts:93`（`permissionRuleValueFromString`，完整的解析加工具名别名）；`src/utils/permissions/permissionRuleParser.ts:55`（`escapeRuleContent`，注释说明转义顺序）

---

### 哨兵占位替换（Sentinel Placeholder Substitution）

**解决的问题**：在字符串处理中，需要临时区分"用户转义的 `*`"和"真正的通配符 `*`"，直接正则替换容易误伤转义的字符。

**核心做法**：用进程空间不可能出现的字符序列（如 `\x00ESCAPED_STAR\x00`）作为哨兵，将转义字符临时替换为哨兵，处理完未转义的字符后，再将哨兵还原为正确的转义序列；哨兵的 `RegExp` 对象模块级定义，只编译一次。

**前置条件**：哨兵字符序列不会出现在合法输入中（null 字节不在命令字符串中）；处理步骤数量有限（哨兵只是临时中转，不需要持久存储）。

**源码证据**：`src/utils/permissions/shellRuleMatching.ts:12`（`ESCAPED_STAR_PLACEHOLDER = '\x00ESCAPED_STAR\x00'`，模块级定义）

## 你能做什么

- **用 `Tool(content)` 格式设计权限规则字符串**，分离工具名和内容模式。工具名是路由键，内容是匹配内容，格式清晰可读，易于版本控制。

- **实现对称的 escape/unescape 函数对**，且转义顺序必须正确：先转义反斜杠，再转义格式分隔符；反转义时逆序。对称性保证序列化/反序列化不丢信息，测试时用 roundtrip 测试（`unescape(escape(s)) === s`）验证对称性。

- **用奇偶计数法（而非简单 `indexOf`）处理转义字符的位置检测**：向前数连续反斜杠数量，偶数个表示字符未被转义（反斜杠成对抵消），奇数个表示字符被转义（最后一个是转义符）。

- **将 legacy 格式的兼容逻辑集中在解析函数入口**（`normalizeLegacyToolName`），不让 legacy 别名泄漏到业务逻辑。解析函数是 legacy 知识的单一终点，业务代码只处理规范化后的名称。

- **用空字节哨兵进行临时字符替换**：在需要区分"已转义的通配符"和"真正的通配符"时，先用不可能出现在合法输入中的字符序列替换已转义的部分，处理完未转义部分后再还原。哨兵的正则表达式模块级定义，避免每次调用都重新编译。
