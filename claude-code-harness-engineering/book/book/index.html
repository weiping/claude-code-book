从 Claude Code 源码提炼的 Harness 工程模式

前言

一次意外的解剖机会

2026 年 4 月，Anthropic 官方发布的 @anthropic-ai/claude-code npm 包中意外留存了一个 60MB 的未压缩调试文件。1902 个源文件、约 51 万行 TypeScript 代码全部暴露——Claude Code v2.1.88 的完整核心逻辑、系统提示词、工具定义一览无余。事件持续约 24 小时后被发现并下架，但代码已在 GitHub 社区广泛传播。

这不是一次普通的安全事故。对于一个以 AI 安全为使命的公司，因工程失误泄露自己的核心代码，讽刺之余也揭示了一个事实：即便是 Anthropic，其 AI 编程工具的竞争力也不在"魔法 AI"，而在工程架构。

源码显示，Claude Code 的成功密码是 60% 模型能力 + 40% 精密工程。那 60% 来自 Claude 模型本身，任何人都可以通过 API 调用获得。而那 40%——提示词的分层缓存、5 种上下文压缩策略、对抗性验证智能体、Feature Flag 驱动的自我精简——恰恰是所有人都可以学习、复制、迁移的部分。

这本书就是对那 40% 的完整解剖。

什么是 Harness 工程

裸用 LLM API，就像把一匹千里马放进没有围栏的草原——它可能往任意方向跑。Harness（马具）是那套缰绳和马鞍，决定了马能稳定朝目标方向跑多远。

在 AI 工程语境里，Harness 指的是围绕 LLM/Agent 搭建的运行框架：提示词结构、工具配置、上下文管理策略、多智能体协作方式、反馈循环设计。Harness 工程就是研究和实践这套框架设计的学问。

过去一年，Anthropic 和 OpenAI 都发表了深度实践文章。Anthropic 用三智能体架构（规划器 + 生成器 + 评估器）在 6 小时内构建出复古游戏制作器；OpenAI 用 Codex 5 个月写出 100 万行代码。他们路径不同，但核心规律一致：一个伟大的模型配一个糟糕的 Harness 会给出糟糕的体验。一个好的模型配一个伟大的 Harness 会给出极致的体验。

从源码提炼模式的方法

这本书不逐行解读代码，不罗列功能。它从 51 万行源码中识别 Harness 的核心子系统，逐一回答七个问题：它解决什么问题、核心数据结构是什么、控制流怎样、并发模型如何、状态存储在哪、错误怎么处理、有什么精巧设计。

每个子系统的分析最终收束为一条可迁移的工程原则，格式为「原则 N.N：[原则名] — [一句话阐述]」。全书 16 章共提炼了数十条这样的原则，最终在第 16 章综合为 7 条核心原则。

如何阅读本书

结构导览。 全书按架构分层自上而下组织，8 篇 16 章：

篇章节主题
第一篇：全景第 1 章全局架构分层和 Harness 控制层的完整边界
第二篇：启动与循环第 2-4 章启动初始化、QueryEngine 主循环、单轮执行流水线
第三篇：上下文工程第 5-6 章系统提示词分层缓存、Skills 懒加载
第四篇：工具编排第 7-8 章buildTool 工厂、Plan Mode 先规划后执行
第五篇：护栏与扩展第 9-10 章权限渐进谱系、Hooks 事件系统
第六篇：记忆管理第 11-12 章5 种压缩策略、forkedAgent 上下文隔离与记忆架构
第七篇：多智能体第 13-14 章GAN 式生成/验证分离、Coordinator 并行协调
第八篇：演进第 15-16 章Feature Flag 自我精简、7 条通用原则

阅读路径。 各篇相对独立，但有前置依赖关系——每章开头标注了前置章节。如果你只想理解某一个问题（如"如何设计上下文压缩"），可以直接跳到对应章节，按前置依赖回溯即可。

术语约定。 全书使用的核心术语及其在 Claude Code 语境下的精确定义，见附录 A 术语表。首次出现的技术术语标注英文原文，如"死代码消除（DCE, Dead Code Elimination）"。

推断标注。 无源码直接依据的结论标注「（推断）」。有源码锚点的结论不标注——锚点本身就是证据。

这本书的最高目标不是教你怎么复制 Claude Code 的实现，而是让你在任何 LLM 应用中回答同一个问题：什么应该由模型管理，什么必须由 Harness 控制？ 89 个 Feature Flag 中的每一个都在等待模型变得足够好的那一天。你的 Harness 也应该如此。
