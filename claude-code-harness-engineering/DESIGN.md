# 从 Claude Code 源码提炼的 Harness 工程模式

## 一、核心主张

Claude Code 不是一个简单的 Claude API 封装。它是 Anthropic 围绕 Claude 模型构建的完整 Harness 控制层——涵盖提示词管理、工具编排、记忆系统、权限护栏、多智能体协调、上下文压缩等所有核心组件。它是目前公开可见的最完整的 AI Harness 工程实现。

本书的立场很明确：**不教你使用 Claude Code，而教你像 Anthropic 工程师一样思考 Harness 设计。**

现有关于 Harness 工程的讨论——无论是 Anthropic 的《Effective Harnesses for Long-Running Agents》，还是 OpenAI 的百万行零手写代码实验——都停留在模式描述层面。"三智能体架构"好，"上下文重置"重要，"Sprint 合约"有效。但模式背后的工程权衡是什么？什么条件下模式会失效？模型能力增强后哪些脚手架应该拆除？这些问题，只有源码能回答。

本书从 Claude Code v2.1.88 的 ~51 万行 TypeScript 源码中提炼 Harness 工程的核心原则。每一条原则都有源码锚点——具体的路径和行号——但全书不展示代码片段。代码是原材料，原则才是产物。读完这本书，你将理解 Anthropic 工程师在"什么应该由模型管理，什么必须由 Harness 控制"这个问题上的每一个具体作答，并能将这些决策框架应用于你自己构建的 AI Agent 应用。

## 二、全书结构

### 前言：打开黑盒的钥匙

前言不纳入章节编号，独立成文，覆盖四个主题：

**主题一：源码泄露——一次意外的解剖机会**

2026 年 4 月，Anthropic 官方发布的 `@anthropic-ai/claude-code` npm 包中意外留存一个 60MB 未压缩调试文件。1902 个源文件、51 万行 TypeScript 代码全部暴露——Claude Code v2.1.88 的完整核心逻辑、系统提示词、工具定义一览无余。事件持续约 24 小时后被发现并下架，但代码已在 GitHub 社区广泛传播。

这不是一次普通的安全事故。对于一个致力于 AI 安全的公司，因工程失误泄露自己的核心代码，讽刺之余也揭示了一个事实：**即便是 Anthropic，其 AI 编程工具的竞争力也不在"魔法 AI"，而在工程架构。** 源码显示，Claude Code 的成功密码是 60% 模型能力 + 40% 精密工程——而那 40%，恰恰是所有人都可以学习、复制、迁移的部分。

**主题二：什么是 Harness 工程**

裸用 LLM API，就像把一匹千里马放进没有围栏的草原，它可能往任意方向跑。Harness（马具）是那套缰绳和马鞍，决定了马能稳定朝目标方向跑多远。在 AI 工程语境里，Harness 指的是围绕 LLM/Agent 搭建的运行框架：提示词结构、工具配置、上下文管理策略、多智能体协作方式、反馈循环设计。Harness 工程就是研究和实践这套框架设计的学问。

过去一年，Anthropic 和 OpenAI 都发表了深度实践文章。Anthropic 用三智能体架构（规划器 + 生成器 + 评估器）在 6 小时内构建出复古游戏制作器；OpenAI 用 Codex 5 个月写出 100 万行代码。他们路径不同，但核心规律一致：**一个伟大的模型配一个糟糕的 Harness 会给出糟糕的体验。一个好的模型配一个伟大的 Harness 会给出极致的体验。**

**主题三：从源码提炼模式的方法**

本书的写作方法：不逐行解读代码，不罗列功能，而是从 51 万行源码中识别 Harness 的核心子系统，逐一回答七个问题——它解决什么问题、核心数据结构是什么、控制流怎样、并发模型如何、状态存储在哪、错误怎么处理、有什么精巧设计。每个子系统的分析最终收束为一条可迁移的工程原则。原则有源码锚点，但代码不进入正文。

**主题四：如何阅读本书**

- 本书采用投影师风格，零代码展示，源码仅作为隐含锚点出现
- 全书按架构分层自上而下组织：从全局鸟瞰到启动层，从核心循环到各子系统，最后收束为通用原则
- 各篇相对独立，但有前置依赖关系（每章标注了前置章节）
- 每章结尾提炼 2-3 条工程原则，格式为「原则 N.N：[原则名] — [一句话阐述]」
- 附录术语表可随时查阅

## 四、排除范围

1. **Stub/未实现模块**：`assistant/`、`server/`、`ssh/`、`proactive/`、`bridge/` 中标注为 stub 的文件不纳入分析
2. **Bash AST 解析器**：`utils/bash/` 中的 AST 相关代码为技术实现细节，不提炼 Harness 模式
3. **功能使用教程**：不教读者如何安装、配置和使用 Claude Code
4. **UI 组件分析**：`components/` 下 144 个 Ink 组件的渲染逻辑不纳入（除非涉及权限交互流程）
5. **与 Harness 无关的代码**：`vim/`、`voice/`、`outputStyles/` 等辅助功能不纳入
6. **背景知识**：前四章简要介绍 Harness 工程核心概念，正文各章不再重复基础概念，假设读者已了解 LLM、Agent、工具调用等
7. **源码重建细节**：本项目是从编译二进制 + source map 重建的，重建过程本身不是本书主题

## 五、写作风格约定

写作风格：`projector`
风格选择理由：本书的目标读者是构建 AI Agent 应用的工程师，他们需要的是从真实源码中提炼的设计原则和决策框架，而非代码导读。投影师风格让源码退居幕后作为锚点，前台展示的是可迁移的工程智慧——这正是本书区别于源码分析文档的核心价值。

格式规则：

1. **源码引用格式**：`相对路径:行号` 或 `相对路径:函数名`，仅在需要验证时出现，不展开为代码展示
2. **语言规则**：中文正文 + 英文技术术语保留原文（如 `QueryEngine`、`Prompt Cache`、`forkedAgent`）
3. **推断标注**：无源码直证的结论注明「（推断）」
4. **零代码展示**：全文不出现代码块。源码细节通过文字描述、架构图（Mermaid）和流程图表达
5. **原则驱动**：每个技术分析最终收束为一条可迁移的工程原则，格式为「**原则 N.N：[原则名]** — [一句话阐述]」
6. **跨章引用**：用"详见第 X 章"代替重复讲解，保持每章的信息密度
7. **图表**：使用 Mermaid 绘制架构图和状态机，用于展示系统结构和数据流
